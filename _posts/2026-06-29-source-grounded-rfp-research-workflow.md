# 도입 (왜 이 글)

LLM(대규모 언어 모델)은 사업계획서 초안을 놀라울 정도로 빠르게 만들어낸다. 섹션 구조도 그럴싸하고, 주요 기술 용어도 맥락에 맞게 들어간다. 그런데 막상 제출 직전, 검토자가 "이 수치 출처가 어디예요?"라고 물어보는 순간 손이 멈춘다.

이 글은 LLM이 만든 제안서 초안을 **방어 가능한 기술계획서**로 바꾸기 위해 내가 직접 만든 검증 워크플로우를 정리한 회고다. 특별한 도구 없이, 항목별 맵 하나로 작성 속도를 크게 끌어올릴 수 있었다.

---

## LLM 초안이 위험한 이유: 자신감 있는 허구

LLM은 그럴듯한 문장을 쓰도록 훈련되어 있다. 문제는 **사실성(factuality)보다 유창성(fluency)이 먼저** 최적화된다는 점이다.

실제로 AI가 만든 초안에는 다음 세 가지 유형의 위험한 주장이 섞여 있다:

| 유형 | 예시 | 실제 위험 |
|------|------|-----------|
| 공개 출처 있음 | 클라우드 서비스 SLA 99.9% | 버전·시점이 틀릴 수 있음 |
| 내부 증빙 필요 | "보유 모델 정확도 93%" | 회사 측정 기준이 없으면 허위 |
| 출처 불명 | "글로벌 도입 사례 다수" | 그냥 지워야 함 |

이 세 유형을 구분하지 않고 그대로 제출하면, 심사·평가 단계에서 신뢰를 잃는다. 특히 정부 R&D 과제처럼 심사위원이 실제 검증을 요구하는 문서일수록 치명적이다.

---

## 핵심 도구: 항목별 작성준비 맵

초안을 고치는 것보다 **항목별로 근거 위치를 먼저 정리하는 것**이 훨씬 빠르다. 나는 아래 4개 컬럼으로 구성된 맵을 스프레드시트 한 장에 만들었다.

```
| 섹션     | 근거 위치              | 검증된 내용              | 작성 제안                  | 회사확인 필요        |
|----------|------------------------|--------------------------|----------------------------|----------------------|
| 3-1 목표 | AWS 공식 문서 (2024)   | SageMaker 파인튜닝 지원  | "SageMaker 기반 파인튜닝"  | -                    |
| 3-2 기술 | 내부 PoC 보고서        | 응답속도 1.2s 달성       | 수치 직접 기입             | PoC 결과 확인 필요   |
| 4-1 실적 | -                      | -                        | -                          | 유사 프로젝트 수주액 |
| 4-3 인력 | 공개 프로필 (LinkedIn) | 핵심 인력 경력 확인됨    | 약력 표 삽입               | -                    |
```

이 맵을 한 번 채우고 나면, 각 섹션에서 **무엇을 쓰면 되는지, 어디에 근거가 있는지**가 명확해진다. 빈 칸이 곧 "오늘 확인해야 할 목록"이 된다.

---

## Claim 분류 기준: 세 가지 버킷

초안의 모든 주장(claim)을 세 버킷으로 분류하는 게 핵심이다.

**버킷 1 — 공개 출처 확인됨**
공식 문서, 논문, 뉴스 기사로 검증 가능. URL과 접근 날짜를 각주에 남긴다. 수치는 인용 시점의 버전을 명시한다.

**버킷 2 — 내부 증빙 필요**
회사가 보유한 실험 결과, 수주 실적, 특허, PoC 보고서로만 뒷받침되는 내용. "작성 가능하지만 담당자 확인 후"로 표시한다.

**버킷 3 — 사용 금지**
출처를 찾을 수 없거나, 찾으려면 비용이 너무 큰 경우. 초안에서 삭제하거나, "일반적으로 알려진 사실"로 약화시킨다.

LLM 초안에서 버킷 3에 해당하는 문장이 생각보다 많다. 특히 "글로벌 XX개사 도입", "정확도 YY% 향상" 같은 구체적 수치는 출처를 바로 찾을 수 없으면 삭제가 안전하다.

---

## 실무에서 배운 것: 초안은 목차다

LLM이 만든 초안을 완성된 문서로 보지 않고 **구조화된 목차**로 보는 순간, 검증이 편해졌다. 초안은 "이런 항목이 필요하다"는 신호이고, 실제 내용은 출처에서 끌어와야 한다.

작업 순서를 정리하면 이렇다:

1. LLM으로 섹션 구조 초안 생성
2. 각 섹션의 핵심 claim 추출
3. 항목별 맵에 claim → 버킷 분류 → 근거 위치 기입
4. 버킷 2·3은 회사 확인 목록으로 분리
5. 버킷 1만 먼저 본문에 작성
6. 확인 완료된 버킷 2 내용 추가

이 흐름을 따르면 "AI가 쓴 내용을 그대로 믿었다가 지적당하는" 상황을 피할 수 있다.

---

## 마치며

AI는 사업계획서를 대신 써주는 도구가 아니라, **작성 준비를 빠르게 해주는 도구**다. 초안을 그대로 믿으면 심사 테이블에서 무너지고, 처음부터 쓰려고 하면 시간이 너무 걸린다.

항목별 맵 하나로 "오늘 쓸 수 있는 것"과 "확인이 더 필요한 것"을 분리하는 습관이 생기면, 다음 제안서부터는 훨씬 빠르고 단단하게 완성할 수 있다. LLM은 초안을 주고, 검증은 내가 한다 — 이 역할 분리가 핵심이다.
