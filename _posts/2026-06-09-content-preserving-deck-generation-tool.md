# 도입 (왜 이 글)

AI 도구에 "이 Markdown으로 발표자료 만들어줘"라고 요청했을 때, 결과물을 펼쳐보면 내용이 절반쯤 증발해 있는 경험을 해본 적 있을 것이다. 긴 설명은 한 줄로 요약되고, 코드 블록은 사라지고, 표는 "데이터 참고"로 대체된다. AI가 **슬라이드답게** 만들려고 자의적으로 정리해버린 것이다.

나는 이 문제를 근본적으로 다르게 접근하기로 했다. 요약·삭제를 허용하는 게 아니라, 원본 Markdown의 모든 구조와 내용을 보존하는 것을 **품질 기준의 최우선**으로 두는 툴을 설계하기 시작했다.

## 왜 PPTX와 HTML을 따로 분리했나

처음에는 하나의 파이프라인으로 두 형식을 모두 뽑으려 했다. 그런데 두 형식이 요구하는 것이 근본적으로 달랐다.

| 형식 | 최우선 가치 | 허용 트레이드오프 |
|------|------------|------------------|
| PPTX | 정확한 구조 변환, 내용 유실 방지 | 디자인 자유도 낮음 |
| 반응형 HTML | 풍부한 인터랙션, 레이아웃 유연성 | 변환 과정에서 더 많은 질문 필요 |

PPTX는 "PowerPoint" 형식 — Microsoft Office 발표 파일 포맷 — 으로, 기업 환경에서 공유·편집이 필요할 때 쓰인다. 여기서는 **변환 정확도**가 생명이다. 반면 HTML 슬라이드는 브라우저에서 바로 열리는 발표자료로, 애니메이션이나 레이아웃 실험이 가능하지만 "어떤 테마를 쓸지", "코드 하이라이트 색상은?" 같은 디자인 결정이 훨씬 많이 개입된다.

이 둘을 하나의 파이프라인에 넣으면 PPTX의 엄격한 검증 로직이 HTML의 유연한 실험과 충돌한다. 그래서 완전히 분리했다.

## content-preserving 변환의 핵심 규칙

**content-preserving** — 내용 보존 우선 — 이란 말을 툴 이름에 넣을 정도로, 이 원칙이 설계 전체를 지배한다. 구체적으로는 다음 규칙 하나로 요약된다.

```
Markdown heading structure → 1:1 또는 1:N slide split, never N:1 merge
```

H2 하나는 슬라이드 한 장 이상으로만 펼쳐진다. 절대로 여러 H2를 하나의 슬라이드로 합치지 않는다. H2가 20개면 최소 20장의 슬라이드가 나온다. 내용이 많아 한 장에 못 들어가면 자동으로 분할(split)한다.

이 규칙이 지켜지는지 자동 검증하는 로직도 핵심이다. 변환 후 다음 항목이 원본과 비교된다:

- **H2/H3 헤딩** 수와 텍스트 일치 여부
- **코드 블록** 개수와 언어 레이블 보존 여부
- **표** 행/열 수 일치 여부
- **긴 문장** (N자 이상) 축약 없이 포함됐는지

누락이 발견되면 변환을 중단하고 경고를 출력한다. 조용히 넘어가지 않는다.

## 출력 구조와 design.md 스펙

결과물은 프로젝트 아래 고정된 경로에 떨어진다:

```
<project>/_decks/<slug>/deck.pptx
<project>/_decks/<slug>/index.html
```

슬라이드마다 스타일을 따로 고민하면 일관성이 무너진다. 그래서 스타일 스펙을 별도 파일로 분리했다:

```
_design_styles/<slug>/design.md
```

**`design.md`** — 이 파일에는 폰트, 색상 팔레트, 여백 규칙, 코드 블록 테마 같은 디자인 결정이 한곳에 모인다. 툴은 콘텐츠를 먼저 파싱한 뒤, 콘텐츠의 성격(기술 문서인지, 스토리텔링인지, 데이터 중심인지)을 분석해 어울리는 `design.md` 프리셋을 추천한다. 사람이 확정하면 그 스펙대로 렌더링한다.

이렇게 하면 콘텐츠 변환 로직과 스타일 결정 로직이 완전히 분리된다. 스타일을 바꾸고 싶을 때 `design.md`만 수정하고 재실행하면 된다.

## 마치며

발표자료 생성 툴을 만들면서 가장 많이 배운 점은 "AI가 알아서 잘 요약해주겠지"라는 기대를 설계 단계에서 버려야 한다는 것이다. 자동 요약은 편리하지만, 발표자가 의도한 맥락과 디테일이 조용히 사라지는 대가를 치른다.

content-preserving 접근은 더 많은 검증 로직을 요구하고 때로는 슬라이드 한 장이 넘쳐흐르게 만들기도 한다. 하지만 그 불편함은 발표자가 직접 조율해야 할 창의적 결정이지, 툴이 임의로 해결해서는 안 되는 문제다. 툴의 역할은 내용을 지키는 것, 그 이상도 이하도 아니다.
