# 도입 (왜 이 글)

사업계획서에 들어가는 정량지표를 직접 설계하기 부담스러울 때, 가장 빠른 선택은 AI에게 초안을 맡기는 것이다. 나도 그렇게 했다. 문제는 그 다음이었다. AI가 뽑아준 수치를 다시 AI로 검토하다 보니, 잘못된 가정을 서로 강화하는 루프에 빠질 뻔했다.

이 글은 그 과정에서 내가 직접 밟은 함정들, 그리고 빠져나오면서 정리한 검증 워크플로를 공유한다.

## AI가 제안한 수치, 첫 인상은 그럴듯했다

처음 AI가 제안한 지표 중 하나는 이런 형태였다.

```
Spearman >= 0.80
```

Spearman(스피어만 순위 상관계수, rank correlation coefficient)은 두 변수 간의 단조적 관계를 측정하는 지표다. AES(Automated Essay Scoring, 자동 에세이 채점) 분야에서도 쓰이긴 한다. 숫자도 0.80이면 꽤 엄격해 보인다. 그래서 처음엔 그냥 넘어갔다.

그런데 문서 전체를 읽다 보니 뭔가 어긋났다. 이 지표가 적용되는 출력 유형이 내가 설계한 시스템과 달랐다.

## 숫자가 아니라 '어떤 출력에 쓰는 지표인가'를 봐야 한다

AES 분야의 표준 지표는 크게 두 방향이 있다.

| 지표 | 적합한 출력 유형 | 특징 |
|------|-----------------|------|
| Spearman | 순위형 점수 | 분포 왜곡에 강하나, 점수 간격 무시 |
| Pearson | 연속형 점수 | 선형 관계 측정, 정규 분포 가정 |
| QWK (Quadratic Weighted Kappa) | 등급형 채점 | 불일치 패널티가 거리에 비례 |

내 시스템은 0~100점 연속 점수를 예측하는 구조였다. Spearman은 순위만 보기 때문에 50점과 51점의 차이를 무시한다. 이 맥락에선 Pearson이 맞는 선택이었다.

```
변경 전: Spearman >= 0.80
변경 후: Pearson >= 0.70
```

수치가 낮아 보이지만, 오히려 더 엄격해졌다. Pearson 0.70은 실제 점수 간격까지 반영한 기준이고, Spearman 0.80은 순위만 맞으면 통과할 수 있었기 때문이다. 표면적인 수치만 보면 기준이 후퇴한 것처럼 느껴지는데, 바로 이 착시가 함정이다.

## AI가 제안한 수치도 출처 없이 확정하면 안 된다

두 번째 함정은 재현율(recall) 지표에서 만났다. AI가 초안에서 제안한 수치는 "재현율 100%"였다. 언뜻 보면 야심차 보이지만, LLM 기반 시스템에서 temperature를 제어하지 않으면 동일 입력에도 출력이 달라질 수 있다. 달성 가능한 목표가 아니었다.

이 수치를 검토할 때 나는 AI에게 재검토를 맡기지 않았다. 대신 직접 논문과 공식 문서를 뒤졌다. 그리고 현실적인 기준을 아래와 같이 재정의했다.

```
변경 전: 재현율 100%
변경 후: 재현율 >= 95% + temperature=0 + caching
```

`temperature=0`은 LLM 추론의 확률적 샘플링을 끄는 설정이다. `caching`은 동일 입력에 대해 캐시된 응답을 재사용해 출력 일관성을 보장한다. 이 두 조건을 명시하지 않으면, 재현율 95%도 사실상 측정 불가능한 목표가 된다.

AI가 뽑은 숫자도 단일 출처이거나 추측 기반일 수 있다. 표준 지표를 다루는 영역이라면 반드시 1차 자료(논문, 공식 문서)와 교차 검증해야 한다.

## '변경 전 → 변경 후 → 사유' 삼단 포맷을 남겨야 하는 이유

지표를 수정하고 나서 가장 잘한 결정은, 변경 결과만 새 표로 만드는 게 아니라 변경 이력을 문서화한 것이었다.

| 항목 | 변경 전 | 변경 후 | 변경 사유 |
|------|---------|---------|-----------|
| 상관지표 | Spearman >= 0.80 | Pearson >= 0.70 | 연속 점수 예측 구조에 적합한 지표로 교체 |
| 재현율 | 100% | >= 95% + 조건 명시 | LLM 특성상 확률적 출력, 재현 조건 없이 측정 불가 |

이 포맷이 있으면 리뷰 과정에서 "왜 수치를 낮췄어요?"라는 질문에 즉시 답할 수 있다. 변경 사유 없이 결과만 보면, 기준이 후퇴한 것처럼 보이기 쉽다. 설득의 부담은 문서가 대신 지게 해야 한다.

덧붙여 실용적인 팁 하나: HWP 파일로 된 원본 계획서를 AI에 넣어 검토할 때는 먼저 텍스트로 변환하는 것이 좋다.

```bash
hwp5txt input.hwp > output.txt
```

`hwp5txt`는 HWP 파일을 평문 텍스트로 추출해주는 CLI 도구다. 변환된 텍스트를 기반으로 AI에 요청하면 원본 포맷에 구애받지 않고 내용 전체를 처리할 수 있다.

## 마치며

AI가 만든 지표를 다시 AI로 검증하는 워크플로는 분명히 빠르다. 하지만 두 AI가 같은 편향을 공유한다면, 오류는 더 단단하게 굳어버린다.

이 경험 이후로 나는 AI 산출물을 검토할 때 세 가지를 먼저 확인한다.

1. 이 지표는 어떤 출력 유형에 맞는 것인가?
2. 제안된 수치의 출처는 1차 자료인가, 추측인가?
3. 변경 이력을 삼단 포맷으로 남길 수 있는가?

AI는 초안을 빠르게 만들어주지만, 그 초안의 타당성을 판단하는 일은 여전히 사람의 몫이다.
