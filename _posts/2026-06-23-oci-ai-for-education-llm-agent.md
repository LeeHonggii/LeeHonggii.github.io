# 도입 (왜 이 글)

교육 도메인에서 LLM Agent를 만들겠다고 마음먹는 순간, 생각보다 많은 질문이 한꺼번에 쏟아진다. 어떤 모델을 쓸지, 학습 데이터를 어디에 올릴지, 검색은 어떻게 붙일지, 그리고 학생 데이터를 외부로 보내도 되는지.

나는 최근 OCI(Oracle Cloud Infrastructure) 기반 교육 LLM Agent 과제를 준비하면서 이 질문들을 공식 문서 중심으로 하나씩 확인했다. 이 글은 그 조사를 정리한 학습 노트다. 정답을 제시하기보다, **초기에 반드시 점검해야 할 갈림길**을 짚는 데 집중한다.

---

## 1. 모델 전략: 관리형 파인튜닝 vs 자체 서빙

OCI Generative AI(이하 GenAI) 서비스는 두 가지 경로를 제공한다.

**관리형 파인튜닝(Managed Fine-Tuning)** — OCI가 제공하는 베이스 모델(Cohere Command 계열 등)에 내 데이터를 올려 파인튜닝하는 방식. 인프라 관리는 OCI가 맡고, 나는 데이터와 하이퍼파라미터만 신경 쓴다.

```jsonl
{"prompt": "다음 학생 답안을 평가하라: ...", "completion": "이 답안은 핵심 개념을 포함하고 있으나..."}
```

학습 데이터는 위와 같은 JSONL 포맷으로 준비한다. 단순하지만, **데이터가 OCI 테넌시 내부에서만 처리되는지** 계약·문서 레벨에서 먼저 확인해야 한다.

**오픈모델 자체 서빙(Self-Serving)** — LLaMA, Mistral 계열을 OCI Compute(GPU) 위에 직접 올리는 방식. LoRA(Low-Rank Adaptation, 경량 파인튜닝 기법)로 도메인 특화 학습이 가능하고, 모델 가중치가 내 인프라에만 존재한다.

```
# LoRA fine-tuning 예시 흐름
base_model = "meta-llama/Llama-3-8B"
lora_rank   = 16
target_modules = ["q_proj", "v_proj"]
```

두 방식의 차이를 한 줄로 정리하면:

| 기준 | 관리형 파인튜닝 | 오픈모델 자체 서빙 |
|---|---|---|
| 운영 복잡도 | 낮음 | 높음 |
| 한국어 성능 | 모델 의존 | 모델 선택 자유 |
| 데이터 레지던시 | OCI 정책 확인 필요 | 완전 통제 가능 |
| 비용 구조 | 호출량 기반 | GPU 상시 비용 |

교육 데이터처럼 민감도가 높은 경우, 자체 서빙이 데이터 통제 측면에서 유리하지만 GPU 운영 부담이 따른다.

## 2. RAG 설계: 벡터 검색만으론 부족하다

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 LLM이 외부 문서를 참조해 답변을 생성하는 구조다. 교육 Agent에서 RAG를 붙일 때 벡터 검색만 설계하면 곧 벽에 부딪힌다.

**벡터 인덱스 설계** — OCI OpenSearch 또는 Oracle Database 23ai의 AI Vector Search를 활용할 수 있다. 인덱스 생성 시 HNSW(Hierarchical Navigable Small World, 근사 최근접 이웃 그래프 알고리즘)와 코사인 유사도를 조합하는 것이 일반적이다.

```sql
-- Oracle DB 23ai 예시
CREATE VECTOR INDEX idx_doc_vec ON documents (embedding)
  ORGANIZATION INMEMORY NEIGHBOR GRAPH
  DISTANCE COSINE
  WITH TARGET ACCURACY 95;
```

**한국어 형태소 검색** — 벡터 검색은 의미 기반이라 완전 일치(키워드) 검색에 약하다. OpenSearch의 `nori` 토크나이저(한국어 형태소 분석기)를 함께 붙여 하이브리드 검색으로 구성하면 정밀도가 올라간다.

```json
"analyzer": {
  "korean": {
    "type": "custom",
    "tokenizer": "nori_tokenizer"
  }
}
```

**감사추적(Audit Trail)과 모델 모니터링** — 교육 현장에서는 "이 답변이 어떤 근거로 나왔는가"를 설명할 수 있어야 한다. OCI AI 서비스의 **Model Catalog + Provenance + Monitoring + Audit** 조합으로 모델 버전, 입력·출력 로그, 드리프트(성능 열화) 탐지를 연결해 설계 초기부터 감사 흔적을 남겨야 한다.

## 3. 교육 데이터의 개인정보와 리전 가용성

이 부분이 가장 자주 놓치는 지점이다.

학생 답안, 성적, 학습 이력은 「개인정보 보호법」상 민감 정보에 해당한다. OCI에서 이 데이터를 처리할 때 확인해야 하는 사항은 크게 세 가지다.

1. **리전(Region) 가용성** — OCI의 GenAI 관리형 서비스가 **한국(서울·춘천) 리전에서 제공되는지** 공식 문서로 확인해야 한다. 2025년 기준 일부 GenAI 기능은 US 리전에서만 제공된다. 만약 한국 리전에서 서비스가 없다면, 데이터를 국외로 전송하는 것이 되어 법적 검토가 필요하다.

2. **비식별화(De-identification) 전략** — 파인튜닝 데이터와 RAG 문서에서 학생 이름·학번·학교명 등 직접 식별자를 제거하거나 가명처리한 뒤에 업로드해야 한다. 이 파이프라인을 아키텍처 초기에 설계해두지 않으면 나중에 걷어내기가 매우 번거롭다.

3. **데이터 처리 위탁 계약** — 클라우드 제공사와의 DPA(Data Processing Agreement)에서 교육 데이터의 재학습 사용 여부, 보존 기간, 제3자 제공 조건을 반드시 확인한다.

---

## 마치며

OCI 기반 교육 LLM Agent를 준비하면서 가장 크게 배운 것은, **기술 선택보다 전제 조건 점검이 먼저**라는 사실이다. 모델을 어떻게 파인튜닝할지 고민하기 전에, 그 데이터가 어디서 처리되는지를 확인해야 한다. RAG 벡터 인덱스를 어떻게 최적화할지 고민하기 전에, 한국어 형태소 분석과 감사추적이 함께 설계되어 있는지 봐야 한다.

체크리스트 형태로 요약하면:

- [ ] 관리형 파인튜닝 vs 자체 서빙: 데이터 레지던시 기준으로 선택
- [ ] RAG: 벡터 + 형태소 하이브리드 + 감사추적 함께 설계
- [ ] 한국 리전 GenAI 서비스 가용성 공식 문서 확인
- [ ] 비식별화 파이프라인 아키텍처 초기 반영
- [ ] DPA 내 교육 데이터 처리 조건 검토

이 네 가지 갈림길을 미리 확인하고 나면, 이후 구현 단계에서 되돌아오는 횟수가 훨씬 줄어든다.
