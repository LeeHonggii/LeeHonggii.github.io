# 도입 (왜 이 글)

교육·훈련용 AI 서비스를 기획하다 보면 어느 순간 이런 유혹이 생긴다. "GPT한테 '나는 이런 직무인데 뭐 공부해야 해?'라고 물으면 되는 거 아냐?" 실제로 그렇게 만든 프로토타입은 그럴싸해 보인다. 커리큘럼 목록도 나오고, 학습 순서도 설명해준다.

문제는 **반복 사용했을 때** 드러난다. 학습자가 뭔가를 배우고 돌아와도 Agent는 그걸 모른다. 이전 대화가 없으면 다시 처음부터 추천한다. "진단"이 없으니 "성장"도 측정되지 않는다. 이건 추천 챗봇이지, 교육 시스템이 아니다.

이 글에서는 교육 LLM Agent를 **진단 → 로드맵 → 학습 → 재진단** 구조의 폐루프(closed-loop)로 설계하는 방법을 정리한다.

---

## 1. 폐루프란 무엇인가 — 열린 추천과의 차이

열린 추천(open-loop)은 입력이 들어오면 출력을 내놓고 끝이다. 피드백이 없다.

폐루프(closed-loop, 되먹임 제어 구조)는 출력 결과가 다시 입력으로 돌아온다. 교육 맥락에서는 이렇게 흐른다:

```
[스킬 추출] → [역량 매핑] → [갭 분석]
       ↑                              ↓
[재진단]   ←  [학습 이력 누적]  ← [로드맵 추천]
```

핵심은 "학습자가 어디 있는지"를 시스템이 지속적으로 추적한다는 것이다. LLM은 이 흐름의 특정 계층에서만 쓰인다. 전체 판단자가 아니다.

---

## 2. 진단 계층 — LLM 혼자 못 하는 이유

역량 진단은 직관적으로 "LLM에게 물어보면 되는 것"처럼 느껴지지만, 실제로는 여러 비-LLM 알고리즘이 함께 필요하다.

**스킬 추출(Skill Extraction)**은 JD(Job Description, 직무기술서)나 이력서에서 기술 키워드를 구조화된 형태로 뽑아내는 작업이다. NER(Named Entity Recognition, 개체명 인식) 모델이나 온톨로지 매칭이 LLM보다 일관성이 높다.

**역량 매핑(Competency Mapping)**은 추출된 스킬을 표준 프레임워크(예: O\*NET, 국가직무능력표준 NCS)의 역량 노드에 연결하는 과정이다. 이 구조가 있어야 "갭"을 정의할 수 있다.

**지식 추적(Knowledge Tracing)**은 학습자가 특정 개념을 얼마나 습득했는지를 시계열로 추적한다. DKT(Deep Knowledge Tracing) 같은 모델이 이 역할을 한다.

가장 강력한 도구는 **IRT/CAT**다.

- **IRT(Item Response Theory, 문항반응이론)**: 각 문항의 난이도·변별도를 수치화하고, 학습자의 능력(θ)을 추정한다.
- **CAT(Computerized Adaptive Testing, 컴퓨터 적응검사)**: IRT 기반으로 다음 문항을 동적으로 선택한다. 20문항 대신 8문항으로도 비슷한 정밀도를 낼 수 있다.

이 두 알고리즘이 없으면 진단은 그냥 "퀴즈 맞히기"에 그친다.

---

## 3. 로드맵 추천 계층 — 경로는 설명 가능해야 한다

진단으로 갭(gap, 현재 역량과 목표 역량의 차이)이 정의됐다면, 다음은 "어떤 순서로 채울 것인가"다.

**선수관계 그래프(Prerequisite Graph)**를 먼저 구성해야 한다. "Docker를 배우려면 Linux 기초가 먼저"처럼 학습 콘텐츠 간의 의존 관계를 DAG(Directed Acyclic Graph, 방향 비순환 그래프)로 표현한다.

그 위에서 **Learning-to-Rank**(학습-순위화, 여러 후보 경로를 학습자 프로파일 기준으로 정렬하는 기법)를 적용한다. 학습 이력, 선호 스타일, 목표 직무, 남은 시간 등을 피처로 넣어 최적 경로를 산출한다.

LLM은 이 경로를 **자연어로 설명**하는 역할을 맡는다. 알고리즘이 낸 결과를 "왜 이 순서인지"로 풀어주는 것이다. 추천 자체를 LLM에 맡기면 경로의 논리가 불투명해지고 재현성도 사라진다.

---

## 4. Agent 계층 설계 — RAG + 평가모델 + 휴먼 검토

실제 운영에서 LLM Agent를 안전하게 쓰는 구조는 세 겹으로 구성한다.

```
[학습자 질의]
      ↓
[RAG Retrieval]  ← 콘텐츠 DB, 역량 온톨로지
      ↓
[LLM 생성]
      ↓
[평가 모델 검증]  ← 정답 기준, 루브릭 스코어링
      ↓
[Human-in-the-loop]  ← 이상값·고위험 케이스만 검토
      ↓
[학습자에게 반환 + 이력 누적]
```

**RAG(Retrieval-Augmented Generation, 검색 증강 생성)**는 LLM이 맥락 없이 답하는 것을 막는다. 검증된 콘텐츠만 참조하게 한다.

**평가 모델**은 LLM 출력의 품질을 자동으로 채점한다. 서술형 답변이라면 QWK(Quadratic Weighted Kappa, 이차 가중 카파) 기반 자동채점 모델이 쓰인다. 정량 지표가 있어야 시스템이 자체 개선 루프를 돌릴 수 있다.

**Human-in-the-loop**(사람 참여 루프)는 전수 검토가 아니라 이상값 필터다. 확신도 점수가 낮거나 학습자 불만이 감지된 케이스만 사람이 검토한다.

---

## 마치며

교육 LLM Agent를 제대로 설계하려면 LLM 하나가 아니라, 진단-추천-학습-재진단을 이어주는 알고리즘 스택 전체를 설계해야 한다. IRT/CAT으로 적은 문항에 정밀한 진단을, 선수관계 그래프와 Learning-to-Rank로 설명 가능한 로드맵을, RAG + 평가모델 + 휴먼 검토로 안전한 Agent를 구성하는 것이 핵심이다.

LLM은 이 폐루프의 "언어 레이어"다. 전체 지능이 아니라, 사람과 시스템을 잇는 인터페이스. 그 역할을 명확히 잡는 순간, 교육 AI는 챗봇에서 진짜 학습 시스템으로 바뀐다.
