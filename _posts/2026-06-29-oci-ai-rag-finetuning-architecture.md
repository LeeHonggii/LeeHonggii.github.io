# 도입 (왜 이 글)

LLM Agent 구현 계획을 세우다 보면 "어떤 모델 쓸까?"보다 훨씬 먼저 결정해야 할 것들이 있다. 리전(Region), 데이터 주권, 파인튜닝 방식, RAG 파이프라인 설계. 이걸 늦게 발견하면 설계를 통째로 갈아엎어야 한다.

최근 교육·훈련 도메인의 LLM Agent 과제를 준비하면서 클라우드 관리형 AI 서비스들을 직접 뜯어봤다. 나중에 또 비슷한 작업을 할 것 같아 학습 노트 형태로 남긴다.

## 리전부터 확인해야 데이터 주권 리스크가 보인다

클라우드 플랫폼의 관리형 생성 AI(Managed Generative AI) — PaaS 형태로 모델을 API처럼 호출하는 방식 — 는 모든 리전에서 제공되지 않는다. "서비스는 쓸 수 있다"는 말이 사실이어도, 원하는 모델이나 파인튜닝 기능이 특정 리전에만 열려 있는 경우가 흔하다.

국내 공공·교육 과제라면 학습 데이터나 개인정보가 국외 서버로 전송되는 순간 개인정보보호법 이슈가 생긴다. 서비스 도입 전에 반드시 확인할 체크리스트:

- 원하는 모델(예: Llama 3, Cohere Command R+)이 국내 또는 국내 인접 리전에 배포되어 있는가?
- 파인튜닝(Fine-tuning) 작업도 같은 리전에서 실행되는가?
- 추론(Inference) 엔드포인트가 국내 리전에서 제공되는가?

리전이 맞지 않으면 아무리 좋은 모델도 도입하기 어렵다. 모델 선택보다 먼저 확인해야 할 것이 바로 이 부분이다.

## 파인튜닝: 관리형 LoRA vs 자체 서빙, 2트랙으로 설계

LoRA(Low-Rank Adaptation, 저랭크 적응)는 전체 모델 가중치를 재학습하지 않고 작은 어댑터 레이어만 학습해 특정 도메인에 모델을 맞추는 기법이다. 메모리·비용 효율이 높아 도메인 특화 파인튜닝의 사실상 표준이 됐다.

클라우드 플랫폼들은 대부분 관리형 LoRA 파인튜닝을 제공한다. 데이터만 올리면 학습부터 엔드포인트 배포까지 처리해준다. 편리하지만 제약도 있다.

| 항목 | 관리형 LoRA | 자체 모델 서빙 |
|------|------------|--------------|
| 초기 세팅 비용 | 낮음 | 높음 (GPU 인프라 필요) |
| 커스터마이즈 자유도 | 낮음 | 높음 |
| 데이터 주권 | 플랫폼 정책 따름 | 직접 통제 가능 |
| 운영 부담 | 낮음 | 높음 |

파인튜닝 데이터는 JSONL(JSON Lines, 한 줄에 하나의 JSON 객체를 담는 포맷) 형식으로 준비하는 게 일반적이다:

```jsonl
{"prompt": "다음 사업계획서의 기술 역량 항목을 채점하시오: ...", "completion": "기술 역량 점수: 85점. 근거: ..."}
{"prompt": "아래 답변의 논리 전개를 평가하시오: ...", "completion": "논리 전개 점수: 72점. 근거: ..."}
```

실전에서는 **2트랙 전략**이 현실적이다. 초기에는 관리형 LoRA로 빠르게 프로토타입을 만들고, 데이터 주권이나 커스터마이즈 요건이 구체화되면 자체 서빙으로 전환한다.

```
Phase 1: Managed LoRA fine-tuning → dedicated hosting endpoint  # 빠른 검증
Phase 2: 자체 GPU 클러스터 + 오픈소스 서빙 프레임워크          # 데이터 주권·비용 최적화
```

## RAG는 벡터 검색 하나가 아니다

RAG(Retrieval-Augmented Generation, 검색 증강 생성)를 "벡터 DB 하나 붙이면 끝"이라고 생각하면 나중에 고생한다. 특히 한국어 도메인에서는.

**한국어 검색기가 별도로 필요하다.** 영어 기준 임베딩 모델은 한국어 형태소 분리가 약해서 "사업계획서"와 "사업 계획서"를 다르게 처리하는 경우도 생긴다. BM25 기반 한국어 키워드 검색기나 한국어 특화 임베딩을 병행해야 recall(재현율, 관련 문서를 빠뜨리지 않고 얼마나 찾아오는가)이 올라간다.

**Hybrid Retrieval(하이브리드 검색)로 설계한다.** 벡터 검색(의미 유사도)과 키워드 검색(정확한 용어 매칭)을 합산하고, Reranker(재순위 모델, 후보 문서를 다시 정렬하는 소형 모델)로 최종 문서를 추려낸다.

```
hybrid retrieval = vector search + keyword search + reranking
```

**정형 데이터는 RDB 필터링 후 RAG에 넘긴다.** 학습자 이력, 채점 기준 같은 구조화된 데이터는 벡터 DB가 아니라 관계형 DB에서 먼저 필터링하는 게 정확도와 속도 모두에서 낫다.

**감사 추적(Audit Trail)을 처음부터 설계에 포함한다.** 교육·공공 도메인은 "AI가 왜 이 점수를 줬는가"를 설명해야 할 상황이 온다. 어떤 문서가 검색됐고, 어떤 컨텍스트가 모델에 들어갔는지 로그로 남겨야 한다. 나중에 추가하려면 파이프라인 전체를 건드려야 하므로 초기 설계에 반드시 포함시켜야 한다.

## 마치며

LLM Agent를 만들겠다고 마음먹으면 제일 먼저 손이 가는 건 모델 선택이다. 그런데 실제로 설계를 시작하면 리전, 데이터 주권, 파인튜닝 전략, RAG 파이프라인이 훨씬 더 복잡한 결정 사항이라는 걸 깨닫게 된다.

"어떤 모델"보다 "데이터가 어디서 처리되는가, 어떻게 검색할 것인가"를 먼저 정의하는 것. 기술 계획을 방어 가능하게 만드는 출발점이 거기에 있다.
