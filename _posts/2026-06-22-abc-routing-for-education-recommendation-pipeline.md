# 도입 (왜 이 글)

처음엔 단순해 보였다. 사용자 입력을 받아 → LLM으로 키워드 추출 → 필터 → 추천. 그런데 실제로 쌓인 로그를 보면 이상한 패턴이 반복됐다. "3학년 2학기 5단원"처럼 학년·단원 정보만 있는 입력이 키워드 검색 경로를 타고, 엉뚱한 결과가 나오는 것이다. 입력의 **의도**가 다른데 같은 파이프라인을 통과시키니 생기는 문제였다.

이 글은 그 파이프라인을 A/B/C 세 가지 입력 유형으로 분기했을 때 무엇이 달라졌는지를 기록한다.

## 문제: 한 줄 파이프라인의 한계

기존 구조는 대략 이랬다.

```
사용자 입력 → LLM 키워드 추출 → DB 필터 → 도구 추천
```

문제는 입력이 세 가지 성격을 가진다는 점이었다.

| 유형 | 예시 | 적합한 경로 |
|------|------|-------------|
| A — 학년·단원 정보만 있음 | "3학년 2학기 분수" | 단원(curriculum unit) 직접 매핑 |
| B — 개념 키워드만 있음 | "등비수열 점화식" | 키워드 풀텍스트 검색 |
| C — 둘 다 있음 | "중2 일차함수 기울기 문제" | 단원 매핑 후 키워드 재랭킹 |

이 세 가지를 같은 필터 경로에 넣으면, A 유형은 키워드가 빈약해서 전혀 다른 단원 결과가 튀어나오고, B 유형은 학년 컨텍스트 없이 상위 학년 도구를 추천하기도 했다.

## 해결: `resolve_input_type` 노드 도입

핵심 아이디어는 **LLM이 추출한 결과에만 의존하지 않고, 룰 기반으로 입력 유형을 먼저 판단하는 노드**를 파이프라인 앞단에 두는 것이다.

```python
# input_type: "A" | "B" | "C"
def resolve_input_type(parsed: dict) -> str:
    has_unit = bool(parsed.get("grade") and parsed.get("chapter"))
    has_keyword = bool(parsed.get("keywords"))
    if has_unit and has_keyword:
        return "C"
    elif has_unit:
        return "A"
    else:
        return "B"
```

이 함수는 LLM 파싱 결과를 받아 명시적인 문자열 타입을 반환한다. 이후 파이프라인은 이 타입 하나로 분기된다.

```
resolve_input_type
    ├── "A" → find_unit_by_grade_semester_chapter → filter_templates
    ├── "B" → keyword_search → filter_templates
    └── "C" → find_unit → keyword_rerank → filter_templates
                          ↓
                  parallel_recommend
```

분기 덕분에 각 경로의 책임이 명확해졌다. A 경로에서는 이런 함수가 핵심이 된다.

```python
async def find_unit_by_grade_semester_chapter(
    grade: int,
    semester: int,
    chapter: str
) -> list[UnitRecord]:
    ...
```

학년·학기·단원명을 명시적 인자로 받으니, 매핑 실패 시 어느 값이 문제인지 바로 보인다. 이전에는 LLM 프롬프트 안에 이 정보가 녹아있어 디버깅이 어려웠다.

## 도구 추천 LLM에도 구조화된 컨텍스트를 넘겨라

분기 구조를 만들고 나서 한 가지를 더 바꿨다. 도구(tool) 추천 단계에서 LLM에 넘기는 프롬프트 컨텍스트였다.

기존엔 사용자 원문만 넣었다.

```
사용자 질문: "일차함수 기울기 문제 풀어줘"
→ 어떤 도구가 적합한가?
```

이 방식은 LLM이 학년 정보를 추론해야 한다. 추론이 틀리면 고등 수준 도구를 중학생에게 추천하는 상황이 생긴다.

구조화 컨텍스트로 바꾸면 이렇게 된다.

```python
context = {
    "grade": 8,           # 중2
    "semester": 1,
    "unit_name": "일차함수",
    "learning_concept": "기울기와 y절편",
    "user_query": "기울기 문제 풀어줘"
}
```

LLM은 이제 추론하지 않아도 된다. 주어진 메타데이터를 기준으로 도구를 고르면 된다. 실험 결과, 학년 불일치로 인한 오추천이 눈에 띄게 줄었다.

## 마치며

파이프라인을 A/B/C로 쪼개는 건 코드 양을 늘리는 결정이다. 분기가 생기면 테스트 케이스도 세 배가 된다. 그런데 그 불편함보다 얻는 게 더 많았다.

- 어떤 경로에서 오추천이 났는지 즉시 알 수 있다.
- 각 경로를 독립적으로 개선할 수 있다.
- LLM에게 "알아서 판단해"라고 떠넘기던 책임을 코드로 명시할 수 있다.

LLM은 유연하지만, 유연함을 파이프라인 설계의 핑계로 쓰면 디버깅이 불가능한 블랙박스가 된다. 입력 유형 분기는 그 블랙박스에 창문을 내는 작업이었다.
