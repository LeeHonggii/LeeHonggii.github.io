# 도입 (왜 이 글)

추천 파이프라인을 처음 짤 때, 나는 모든 입력을 같은 경로에 태웠다. 키워드가 들어오든, "3학년 2학기 3단원"처럼 학년·차시 정보가 들어오든, 둘 다 섞인 질의가 들어오든 — 일단 `validate → extract_keywords → filter_templates → recommend` 순서대로 흘렸다. 처음엔 잘 됐다. 그런데 실제 데이터가 쌓이자 추천 결과가 이상해지기 시작했다. 키워드 검색이 학년 정보에 오염되고, 학년·차시 검색이 엉뚱한 단원으로 매핑됐다. 단일 경로가 문제였다.

## 단일 경로가 왜 오염되는가

파이프라인을 단순화하면 이렇다.

```
입력 → extract_keywords → filter_templates → LLM 추천 → 결과
```

`extract_keywords` — LLM이 입력에서 키워드를 뽑는 단계 — 는 "3학년 2학기 분수의 덧셈"을 받으면 `["분수", "덧셈"]`만 돌려주기도 하고 `["3학년", "2학기", "분수"]`를 돌려주기도 한다. 응답이 일관되지 않다. 그 결과물이 그대로 `filter_templates`에 들어가면, 학년 필터가 걸려야 할 자리에 키워드 필터가 걸리거나 그 반대가 일어난다. LLM 키워드 추출 자체가 비결정적(non-deterministic)이라 단계가 늘어날수록 오염이 누적된다.

핵심 문제: **입력 유형을 구분하지 않고 동일 전략을 적용했다**.

## 입력 유형을 먼저 판별한다 — resolve_input_type

해결책은 파이프라인 앞단에 라우터(router)를 두는 것이다. 입력이 어떤 종류인지 먼저 판별하고, 그에 맞는 후보 생성 전략으로 분기한다.

```
validate → extract_keywords → resolve_input_type → (A/B/C 분기) → filter_templates → parallel_recommend → merge
```

분기 기준은 단순하다.

| 타입 | 조건 | 전략 |
|------|------|------|
| **A** | 학년·학기·단원 정보만 있음 | 커리큘럼 역인덱스로 단원 직접 조회 |
| **B** | 키워드만 있음 | 키워드 매칭으로 후보 필터링 |
| **C** | 둘 다 있음 | A 결과를 1차 필터로, B를 2차 리랭킹에 사용 |

코드로 표현하면 `input_type: "A" | "B" | "C"` 필드 하나가 이후 모든 단계의 전략을 결정한다. 타입 A는 아래처럼 DB를 직접 쿼리해서 후보군을 좁힌다.

```python
async def find_unit_by_grade_semester_chapter(
    school_level: str,
    grade: int,
    semester: int,
    chapter: int
) -> Unit | None:
    # 커리큘럼 역인덱스 기반 직접 조회
    # LLM 키워드 추출 경로를 완전히 우회
    ...
```

타입 A에서는 LLM을 키워드 추출 목적으로 쓸 필요가 없다. 구조화된 정보가 이미 있기 때문이다.

## 도구 추천 AI에도 구조화 컨텍스트를 전달해야 한다

분기를 나눈 것만으로는 충분하지 않았다. 최종 도구·템플릿 추천을 담당하는 LLM에 원문 질의만 넘기고 있었는데, 그 LLM도 학년이나 단원을 모르면 엉뚱한 걸 추천한다.

수정 전:
```python
recommend(query=user_input)
```

수정 후:
```python
recommend(
    query=user_input,
    context={
        "school_level": "middle",
        "grade": 2,
        "semester": 1,
        "chapter": 3,
        "unit_name": "일차방정식",
        "learning_concepts": ["이항", "등식의 성질"]
    }
)
```

이 구조화 컨텍스트(structured context)가 없으면 추천 LLM은 "이항"이 대수(algebra) 맥락인지 다른 맥락인지 스스로 추론해야 한다. 정보를 줄수록 추천 품질이 올라간다는 건 당연하지만, 파이프라인 중간에서 이 정보가 소실되는 경우가 생각보다 많다.

## 마치며

단일 경로를 A/B/C로 나눈 이후 추천 결과의 일관성이 눈에 띄게 좋아졌다. 핵심은 두 가지다.

1. **입력 유형을 파이프라인 앞단에서 판별하고, 후보 생성 전략을 그에 맞게 분리한다.** 하나의 필터 로직이 모든 유형을 처리하려 하면 각 경로가 서로를 오염시킨다.
2. **추천 AI에 넘기는 컨텍스트는 원문만이 아니라 구조화된 메타데이터까지 포함해야 한다.** LLM은 정보가 없으면 추론하고, 추론은 틀린다.

라우팅 로직 자체는 단순한 조건문이지만, 이 분기 하나가 파이프라인 전체 신뢰성을 바꿨다. 복잡한 모델보다 명확한 입력 분류가 먼저다.
