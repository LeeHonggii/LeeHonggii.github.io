# 도입 (왜 이 글)

추천 시스템에서 한국어 질의 정규화를 처음 설계할 때, 나는 꽤 자신이 있었다. "뺄셈이랑 뺄샘은 당연히 같은 거고, 띄어쓰기 정도만 정리하면 되겠지." 그런데 막상 실제 사용자 입력 로그를 열어보니 생각보다 훨씬 복잡한 표면 변형들이 쌓여 있었다.

이 글은 감으로 만든 정규화 규칙이 왜 오탐을 키우는지, 그리고 실제 로그 분석에서 어떤 패턴이 나왔는지를 정리한 학습 노트다.

## 감으로 만든 규칙의 한계

처음에는 몇 가지 규칙을 직관으로 추가했다.

- 공백 제거
- 소문자 통일
- 조사 `을/를`, `이/가` 처리

규칙이 10개쯤 됐을 때 문제가 생겼다. 특정 패턴을 잡으려고 추가한 규칙이 다른 정상 질의를 오탐하기 시작한 것이다. "미분" 과 "미분방정식" 이 같은 버킷으로 묶이는 식으로. 규칙끼리 충돌이 생겨도 어느 로그에서 비롯된 건지 근거가 없으니 디버깅 자체가 막막했다.

근거 없이 쌓은 규칙은 결국 유지보수 부채가 된다.

## 로그를 까봤더니: 실제로 어떤 변형이 많았나

분석 스크립트를 만들어 실제 검색 로그 2104건을 읽어들였다.

```bash
python scripts/analyze_user_inputs_for_normalization.py
```

중복 제거(dedupe)를 적용하자 1849건으로 줄었다. 이 1849건을 변형 유형별로 클러스터링해보니 내 예상과 달랐다.

| 변형 유형 | 예시 | 비율 |
|---|---|---|
| 오타 (자모 혼동) | `뺄샘`, `곱셉` | 약 18% |
| 학년·차시 표현 불일치 | `중2-1`, `중학교 2학년 1학기` | 약 31% |
| 어미·조사 붙임 | `분수를`, `방정식이란` | 약 27% |
| 띄어쓰기 | `소수점 이하`, `소수점이하` | 약 15% |
| 기타 | 단위 혼용, 영한 혼재 | 약 9% |

의미론적 변형보다 **표면 변형(surface variation)** 이 훨씬 많았다. 임베딩(embedding, 텍스트를 벡터로 변환하는 기법)을 도입하기 전에 이 표면 변형들을 먼저 잡는 게 맞겠다는 판단이 섰다.

## 임베딩 없이 어디까지 커버할 수 있나

로그 분석 결과를 바탕으로 3개 레이어를 쌓았다.

**1. Text Normalizer (텍스트 정규화기)**

```python
def normalize_text(s: str) -> str:
    s = s.strip().lower()
    s = re.sub(r"\s+", " ", s)          # 중복 공백 제거
    s = re.sub(r"[^\w가-힣\s]", "", s)  # 특수문자 제거
    return s
```

**2. Alias Map (별칭 매핑)**

오타와 동의어는 alias map으로 관리한다. 규칙이 아니라 데이터로 관리하는 게 핵심이다.

```python
ALIAS_MAP = {
    "뺄샘": "뺄셈",
    "곱셉": "곱셈",
    "중2-1": "중학교 2학년 1학기",
}

def normalize_for_match(s: str) -> str:
    s = normalize_text(s)
    return ALIAS_MAP.get(s, s)
```

**3. Curriculum Reverse Index (교육과정 역인덱스)**

학년·학기·단원 표현은 경우의 수가 많아서 alias map만으로는 부족하다. 대신 교육과정 구조를 역인덱스(reverse index, 검색어→원문 매핑)로 구성하고 substring matching으로 연결했다. `"중2 1학기 방정식"` 처럼 부분 표현이 들어와도 올바른 단원 코드로 떨어지게 된다.

## 마치며

로그를 먼저 분석하고 나니 오히려 해야 할 일이 줄었다. 임베딩 도입을 검토했던 케이스의 상당수가 사실은 단순 오타나 학년 표현 불일치였기 때문이다. 복잡한 해법 전에 데이터를 보는 것, 그게 가장 싼 디버깅이었다.
