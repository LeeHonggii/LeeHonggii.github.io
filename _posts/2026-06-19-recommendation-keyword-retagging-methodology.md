# 도입 (왜 이 글)

콘텐츠에 키워드를 붙이는 일은 쉬워 보인다. 제목 읽고, 핵심 단어 몇 개 뽑으면 끝 아닌가? 그런데 막상 수백 개 콘텐츠의 키워드를 검수하다 보면 이상한 현상을 발견한다. 키워드가 많고 정확한데 추천이 잘 안 된다. 반대로 키워드가 2~3개뿐인데 연관 콘텐츠 묶음이 깔끔하게 잡힌다.

이유는 단순하다. **추천 시스템은 키워드의 정확성이 아니라 키워드의 겹침(overlap)** 을 보기 때문이다. 이 글은 내가 대규모 메타데이터 재검증 작업을 하면서 직접 부딪힌 기준들을 정리한 구현 가이드다.

---

## 키워드는 콘텐츠 혼자 보지 말고 "풀(pool)"과 함께 봐야 한다

처음에는 콘텐츠별로 제목과 설명만 읽고 키워드를 판단했다. 그런데 이 방식은 구조적으로 망가진다.

예를 들어 "Python으로 배우는 데이터 전처리" 강의에 `python`, `데이터전처리`, `판다스활용실습` 세 개를 붙였다고 하자. 세 단어 모두 콘텐츠를 잘 설명한다. 그런데 전체 키워드 풀에 `판다스활용실습`을 쓰는 다른 콘텐츠가 하나도 없다면, 이 키워드는 추천 그래프에서 고립 노드가 된다.

실제로 효과적인 판단 순서는 이렇다:

1. **제목 + 설명** — 콘텐츠가 무엇인지 파악
2. **기존 원본 키워드** — 이미 붙어 있던 키워드 중 유효한 것 재활용
3. **전체 키워드 풀** — 지금 붙이려는 키워드가 다른 콘텐츠와 실제로 겹치는지 확인

세 번째 단계를 빠뜨리면 키워드가 많아도 추천 매칭이 안 된다.

---

## 1회성 키워드는 제거하거나 상위 개념으로 흡수한다

재검증을 하다 보면 어떤 키워드는 전체 풀에서 딱 한 콘텐츠에만 붙어 있다. 예를 들어 `AWS람다트리거설정` 같은 식으로 너무 구체적인 경우다.

이런 키워드는 두 가지 중 하나다:

| 상황 | 처리 방법 |
|------|-----------|
| 상위 개념이 풀에 있다 (`클라우드자동화`) | 상위 개념으로 교체 |
| 상위 개념도 없고 겹치는 콘텐츠도 없다 | 과감히 제거 |

"이 키워드가 정확한가?"가 아니라 **"이 키워드가 다른 콘텐츠와 연결을 만들 수 있는가?"** 가 기준이다. 추천 시스템은 정의 사전이 아니라 연결 그래프로 동작하기 때문이다.

---

## 2~3개 원칙, 시리즈는 일관성, cross-strand도 허용

실제로 운영하면서 정착된 수량 기준은 **콘텐츠당 키워드 2~3개** 다.

- 1개: 연결점이 너무 적다
- 4개 이상: 노이즈가 많아지고 키워드 간 희석이 생긴다
- **2~3개**: 핵심 연결고리만 남고 그래프가 깔끔해진다

시리즈 콘텐츠의 경우 추가 규칙이 있다. 같은 시리즈 내 강의들은 **최소 1개 이상의 공통 키워드**를 유지해야 한다. 그래야 "이 강의를 들었으면 다음 강의도" 패턴이 추천으로 이어진다.

또 하나 흥미로웠던 점은 **cross-strand 키워드**, 즉 서로 다른 카테고리의 콘텐츠를 연결하는 키워드를 허용하는 게 오히려 좋다는 것이다. 예를 들어 `자동화`는 개발 강의에도, 업무효율 강의에도 붙을 수 있다. 이 키워드 하나가 두 카테고리 사이의 다리가 된다.

---

## 검증: result ↔ 원본 mismatch = 0

키워드 작업 후에는 반드시 검증이 필요하다. 내가 쓰는 방식은 단순하다.

배치 작업 결과 파일 구조:

```
rematch_batches/<strand>_batchNN.json          # 입력
rematch_batches/<strand>_batchNN_result.json   # 출력
```

결과 JSON 구조 예시:

```json
{ "cid": "content-001", "keywords": ["AI", "데이터", "자동화"] }
```

이 result 파일의 키워드를 원본 메타데이터의 `keyword_tags` 필드에 덮어쓴 뒤, 원본 버전(v15 등)과 비교해서 **mismatch = 0** 이 되는지 확인한다. mismatch가 남아 있다면 어딘가 누락되거나 잘못 매핑된 항목이 있다는 신호다.

이 검증 단계가 없으면 수백 개를 재검증했다고 해도 실제 시스템에 반영된 데이터는 다를 수 있다.

## 마치며

키워드 태깅은 "이 단어가 이 콘텐츠를 잘 설명하는가"의 문제가 아니다. **"이 단어가 추천 그래프 안에서 연결을 만드는가"** 의 문제다.

LLM을 활용해 메타데이터를 자동 생성할 때도 마찬가지다. 자동 추출 결과를 그대로 쓰는 게 아니라, 전체 키워드 풀과 비교하면서 1회성 키워드를 걸러내고 겹침을 의도적으로 설계해야 한다.

키워드는 적게, 그러나 전략적으로. 그게 추천 시스템에 실제로 작동하는 태깅이다.
