# 도입 (왜 이 글)

콘텐츠 메타데이터에 키워드를 붙이는 작업을 처음 시작했을 때, 나는 본능적으로 "많이 뽑을수록 좋다"고 생각했다. LLM(대형 언어 모델, Large Language Model)은 제목 하나에서도 키워드를 10개씩 뽑아낼 수 있으니까. 그런데 배치 검증 작업을 반복하면서 깨달은 게 있다. **추천 시스템에서 키워드는 검색 인덱스가 아니다.** 키워드는 "이 콘텐츠를 어떤 사용자와 다시 만나게 할 것인가"를 결정하는 매칭 신호다.

---

## 키워드를 '많이' 뽑으면 생기는 문제

LLM에게 태깅을 맡기면 흔히 이런 결과가 나온다.

```json
{
  "cid": "content_0042",
  "keywords": [
    "데이터 파이프라인", "ETL", "배치 처리", "Airflow", "DAG 설계",
    "스케줄러 최적화", "재시도 로직", "멱등성", "운영 자동화", "모니터링"
  ]
}
```

이 중에서 추천 매칭에 실제로 쓸 수 있는 태그는 몇 개일까? `DAG 설계`나 `재시도 로직`은 이 콘텐츠에만 해당하는 너무 세부적인 키워드다. 비슷한 콘텐츠가 10개 있어도 `재시도 로직`이라는 태그를 공유할 가능성은 낮다. 재사용성(reusability)이 없는 태그는 추천 그래프에서 고립된 노드가 된다. 사용자가 어떤 콘텐츠를 보더라도 그 태그를 경유해서 다른 콘텐츠로 이어지지 않는다.

---

## 내가 정립한 기준: 제목·설명·원본 태그를 함께 본다

태그를 붙일 때 세 가지 정보를 동시에 본다.

| 입력 | 역할 |
|------|------|
| 제목 | 콘텐츠의 핵심 의도 |
| 설명(description) | 실제 다루는 범위 |
| 원본 `keyword_tags` | LLM이 이미 뽑은 후보 풀 |

이 셋을 겹쳐보면서 **최소 2~3개의 재사용 가능한 키워드**만 남긴다. 판단 기준은 단순하다. "다른 콘텐츠 5개 이상에서도 이 태그가 붙을 수 있는가?" 이 질문에 Yes라면 남기고, No라면 제외한다.

결과물은 이렇게 간결해진다.

```json
{
  "cid": "content_0042",
  "keywords": ["데이터 파이프라인", "ETL", "운영 자동화"]
}
```

`keyword_tags` 덮어쓰기는 이 검증된 배열로 source-of-truth JSON에 반영한다. 원본을 보존하되, 추천 시스템이 실제로 읽는 필드는 이 정제된 버전이다.

## generic 태그 단독 사용 금지

또 하나의 함정은 너무 넓은 generic(일반적인) 태그다. 실제로 금지 목록을 만들어뒀다.

```
generic 단독 금지: 기초 / 실무 / 이해 / 전략 / 관리 / 분석
```

`분석`이라는 태그가 붙은 콘텐츠가 500개라면, 이 태그는 추천 신호로서 아무 의미가 없다. 노이즈다. 단, generic 태그가 **구체적인 태그와 함께** 쓰일 때는 허용한다. `분석` 단독은 안 되지만, `SQL 분석`이나 `A/B 테스트 분석`은 괜찮다.

반대로, 주제의 경계를 넘는 cross-strand(여러 영역에 걸친) 태그는 오히려 장려한다. 예를 들어 데이터 엔지니어링 콘텐츠에 `팀 협업`이나 `문서화` 태그가 붙는 경우다. 사용자가 기술 주제가 아니라 "함께 일하는 방식"으로 콘텐츠를 찾을 수도 있기 때문이다. 태그 체계는 기술 분류 트리가 아니라 **의미 기반 매칭 네트워크**여야 한다.

---

## 마치며

LLM 기반 태깅 파이프라인을 처음 만들 때는 자동화에 집중하기 쉽다. 모델이 뽑아준 키워드를 그대로 쓰는 게 가장 빠르니까. 하지만 추천 시스템의 품질은 결국 태그의 재사용성과 매칭 밀도에서 갈린다.

핵심을 한 문장으로 정리하면 이렇다.

> **좋은 태그는 콘텐츠를 설명하는 게 아니라, 콘텐츠가 다시 발견될 경로를 만드는 것이다.**

많이 뽑는 것보다 적게, 그러나 재사용 가능하게. 이 원칙 하나가 수천 개 콘텐츠의 추천 품질을 바꾼다.
