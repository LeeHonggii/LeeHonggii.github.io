# 도입 (왜 이 글)

키워드 추출 로직이 틀렸다는 걸 처음 발견했을 때, 나는 가장 빠른 방법을 선택했다. 문제가 된 입력을 그대로 few-shot 예시에 넣었다. 테스트를 돌렸고, 통과했다. "해결됐다"고 생각했다.

그 판단이 틀렸다는 걸 조금 지나서야 깨달았다.

## few-shot은 해당 케이스를 외울 뿐이다

few-shot — LLM(대형 언어 모델)에게 입력-출력 예시를 몇 개 보여주는 방식 — 은 강력하지만, 본질적으로 패턴 매칭이다. 특정 입력 A에 대해 B를 기대한다고 알려주면, A와 충분히 비슷한 입력에는 B를 돌려준다.

문제는 "충분히 비슷한"의 기준을 모델이 결정한다는 점이다.

예를 들어 이런 케이스가 있었다:

```
# 잘못된 분리
입력: "수 모으기 활동"
출력: keywords=["수", "활동"]

# few-shot에 고정한 예시
입력: "수 모으기 활동" → keywords=["모으기"]
```

이 예시 하나로 해당 케이스는 고쳐진다. 하지만 "수 가르기와 모으기" 혹은 "모으기의 개념 이해"는 어떻게 처리될까? 보장이 없다. 모델은 일반 규칙을 배운 게 아니라 그 문장 하나를 외운 것에 가깝다.

## "전체와 부분의 관계 이해"를 키워드로 단정할 수 있을까?

더 까다로운 케이스가 있었다. 학습 목표처럼 쓰이는 문장형 개념이었다:

```
입력: "전체와 부분의 관계 이해"
```

이걸 보면 특정 수학 개념으로 곧장 매핑하고 싶은 충동이 생긴다. "분수"일 수도 있고, "집합"일 수도, "비율"일 수도 있다. 하지만 학년과 단원 맥락 없이는 어떤 매핑도 자의적이다.

여기서 few-shot으로 특정 정답을 고정하면, 같은 표현이 다른 맥락에서 나왔을 때 오히려 잘못된 키워드를 강하게 확신하는 모델이 만들어진다. 올바른 접근은 이렇다:

```json
{
  "query": "전체와 부분의 관계 이해",
  "candidates": ["분수", "비율", "집합"],
  "needs_clarification": true
}
```

확신이 없으면 `needs_clarification: true`를 반환하고 후보 목록을 열어두는 것으로 충분하다. "모르겠다"고 표현하게 만드는 것 자체가 설계의 일부다.

## LLM은 rerank에, 후보 생성은 결정론적 로직에

few-shot의 한계를 경험하면서 구조를 바꿨다.

| 역할 | 담당 |
|---|---|
| 후보 생성 | 정규화 사전, alias 목록, 교육과정 역인덱스(curriculum reverse index) |
| 최종 판별 | LLM이 후보 중에서 고르기 |

rerank — 여러 후보 중 우선순위를 재정렬하는 작업 — 에 LLM을 쓰면 "후보 없이 자유롭게 생성"하는 것보다 훨씬 안정적이다. 탐색 공간이 미리 정해져 있고, LLM이 할 일은 발명이 아니라 판단이기 때문이다. 결정론적(deterministic) 로직은 동일 입력에 동일 후보를 내놓는다. 재현 가능하고 디버깅이 쉽다.

## 마치며

장애 케이스를 few-shot에 하드코딩하는 건 빠른 픽스처럼 보이지만, 테스트를 통과시키는 것과 문제를 해결하는 것은 다르다. 진짜 일반화는 모델이 한 번도 본 적 없는 유사 입력에서도 올바르게 작동해야 한다.

지금은 LLM에게 "정답을 만들어"라고 시키는 대신, "이 후보들 중에 뭐가 맞아?"라고 묻는 쪽으로 프롬프트 전략을 바꿨다. 불확실하면 불확실하다고 표현하게 하고, 확신이 없는 케이스는 사람이 검토하도록 열어뒀다.

LLM을 믿되, 탐색 공간은 코드로 잡는다.
