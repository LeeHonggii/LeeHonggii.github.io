# 도입 — 협업 필터링이 통하지 않는 순간

추천 시스템을 처음 설계할 때 나는 당연히 협업 필터링(Collaborative Filtering, CF)부터 떠올렸다. "이 사람과 비슷한 사람이 봤던 콘텐츠를 보여주면 되지 않나?" 그런데 실제로 데이터를 들여다보니 문제가 있었다. 플랫폼이 아직 초기 단계라 평점 데이터도, 이용 로그도 절대적으로 부족했다. CF는 아이템-유저 행렬이 어느 정도 채워져야 작동하는데, 비어 있는 행렬에 SVD(특잇값 분해)를 돌려봤자 의미 있는 잠재 벡터가 나오지 않는다. 이걸 **콜드 스타트 문제(Cold Start Problem)**라고 부른다.

결국 방향을 바꿨다. 행동 데이터가 없으면 **콘텐츠 자체의 속성**과 **유저의 컨텍스트**로 유사도를 계산하는 콘텐츠 기반 필터링(Content-Based Filtering, CBF)이 현실적인 선택이다. 이 글에서는 그 설계 과정을 정리한다.

## 왜 협업 필터링이 아닌가

두 방식을 나란히 놓으면 선택 기준이 명확해진다.

| 방식 | 원리 | 필요 데이터 | 적용 여부 |
|---|---|---|---|
| 협업 필터링 (CF) | 비슷한 유저의 행동 패턴 참조 | 평점, 클릭, 수강 이력 | 데이터 부족 → 보류 |
| 콘텐츠 기반 필터링 (CBF) | 콘텐츠 속성과 유저 프로필 유사도 | 직무, 직급, 태그, 목표 | **채택** |
| 하이브리드 | CF + CBF 결합 | 둘 다 | 추후 고려 |

CF는 데이터가 쌓이면 강력하다. 하지만 그 전까지는 CBF가 훨씬 안정적이고, 무엇보다 **결과를 해석하고 디버깅할 수 있다**는 점이 좋았다. "왜 이 콘텐츠가 추천됐어?"라는 질문에 명확히 답할 수 있어야 했다.

## 유사도 계산에 사용한 6개 신호

CBF의 핵심은 "무엇을 신호로 쓸 것인가"다. 나는 콘텐츠 속성과 유저 프로필을 대조해 아래 6개 신호를 정의했다.

```
유사도 = Σ (신호_i × 가중치_i)

신호 목록:
  1. 직무 일치도         — 유저 직무 vs 콘텐츠 타깃 직무
  2. 직급 적합도         — 신입/중급/시니어 레벨 매칭
  3. 학습 목표 겹침      — 유저 설정 목표 태그 vs 콘텐츠 학습 목표
  4. 선수 콘텐츠 이수    — 필요 선수 학습 완료 여부 (0 or 1)
  5. 콘텐츠 유형 선호    — 동영상/문서/퀴즈 선호 프로필
  6. 최근 학습 맥락      — 최근 7일 이내 수강 도메인과 유사도
```

각 신호는 0~1 사이 값으로 정규화했고, 가중치 합은 1이 되도록 맞췄다. 처음에는 모든 신호를 동등하게 줬다가, 직무 일치도와 학습 목표 겹침이 실제 적합도에 훨씬 크게 영향을 미친다는 걸 확인하고 가중치를 조정했다.

## 시나리오 기반 가중치 분리

추천 진입점은 단일하지 않다. 신입사원이 온보딩할 때와, 시니어가 자격증 준비를 할 때, 같은 알고리즘을 쓰면 엉뚱한 결과가 나온다. 그래서 **시나리오(Scenario)** 단위로 가중치를 분리했다.

| 시나리오 | 설명 | 가중치 포커스 |
|---|---|---|
| S1 — 신규 입사자 | 기초 직무 지식 습득 | 직무 일치도 ↑, 난이도(신입) ↑ |
| S2 — 직무 심화 | 현재 역할의 심층 학습 | 학습 목표 겹침 ↑, 최근 맥락 ↑ |
| S3 — 자격 취득 | 시험·인증 준비 | 선수 이수 여부 ↑, 콘텐츠 유형 ↑ |

시나리오는 유저 프로필이나 명시적 학습 목표 선택에서 자동으로 판단된다. S1이라면 직무 일치도를 0.4로 높이고 선수 이수 신호는 0.05로 낮추는 식이다. 이렇게 분리하면 **하나의 유사도 함수**로 다양한 맥락에 대응할 수 있다.

## 결정론적 추천 로직의 장점

이 설계는 전통적인 머신러닝 모델 없이 **순수한 규칙 기반(rule-based) + 가중 유사도 합산**으로 동작한다. 처음엔 "너무 단순한 거 아닌가?" 싶었는데, 운영하면서 오히려 장점이 명확했다.

- **디버깅이 쉽다.** 유저가 "왜 이게 추천됐어요?"라고 물으면 각 신호 점수를 로그로 바로 꺼낼 수 있다.
- **빠르다.** 모델 서빙 서버 없이 백엔드 서비스 내부에서 인메모리로 계산 가능하다.
- **조정이 즉각적이다.** 가중치 하나 바꾸면 배포 없이 결과가 달라진다. A/B 테스트 루프가 짧다.

물론 한계도 있다. 유저의 암묵적 선호(실제로 좋아하는데 프로필에 없는 것)는 잡지 못한다. 이건 데이터가 쌓이면 CF와 하이브리드로 보완할 계획이다.

## 마치며

"데이터가 없으면 추천 못 한다"는 생각부터 버렸을 때 길이 보였다. 유저가 자신에 대해 알려주는 정보(직무, 목표, 레벨)와 콘텐츠가 스스로 갖고 있는 속성을 연결하면, 행동 로그 없이도 꽤 합리적인 추천이 가능하다. 물론 이건 영구적인 해법이 아니라 **초기 단계의 현실적인 선택**이다. 다음 스텝은 이 결정론적 점수를 베이스라인으로 놓고, 클릭·완강 데이터가 쌓이면 CF 신호를 하나씩 얹는 것이다. 완벽한 시스템보다 지금 당장 작동하는 시스템이 낫다.
