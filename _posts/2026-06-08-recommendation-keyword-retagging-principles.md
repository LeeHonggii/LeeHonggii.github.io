# 도입 (왜 이 글)

콘텐츠 메타데이터에 키워드를 붙이는 작업을 LLM으로 자동화하면서 처음엔 "많이 뽑을수록 좋다"고 생각했다. 커버리지가 넓어지니까. 그런데 실제로 추천 시스템에 연결해보니 정반대였다. 키워드가 많아질수록 매칭이 흐릿해졌고, 결국 아무것도 제대로 추천하지 못하는 상태가 됐다. 이 글은 그 삽질을 정리한 학습 노트다.

## 문제: LLM은 키워드를 '설명'처럼 뽑는다

LLM에게 "이 콘텐츠의 키워드를 뽑아줘"라고 하면 보통 10개 이상이 나온다. 그리고 그 중엔 이런 것들이 섞여 있다.

- `기초`, `실무`, `이해`, `전략`, `관리`, `분석`

얼핏 보면 무해해 보이지만, 추천 시스템 관점에서 이건 **노이즈**다. 코퍼스(corpus, 전체 콘텐츠 집합) 안에서 "기초"라는 키워드를 가진 콘텐츠가 수백 개라면, 이 키워드는 아무런 변별력이 없다. 사용자가 "기초" 콘텐츠를 찾는 게 아니라 "머신러닝 기초"를 찾는 거니까.

반대로 너무 특수한 키워드도 문제다. `2024년_특정_프레임워크_버그_픽스` 같은 키워드는 딱 그 콘텐츠 하나에만 붙는다. 매칭할 다른 콘텐츠가 없으니 추천에 기여하지 못한다.

## 원칙: 키워드는 '재사용 가능한 교차점'이어야 한다

추천 시스템에서 키워드의 역할은 **콘텐츠 간 연결고리**다. A 콘텐츠와 B 콘텐츠가 같은 키워드를 공유할 때, 시스템은 둘을 연관 콘텐츠로 묶을 수 있다. 그러려면 키워드가 여러 콘텐츠에 걸쳐 재사용될 수 있어야 한다.

내가 정착한 원칙은 세 가지다.

1. **목표는 2-3개**: 적게 뽑되, 그 콘텐츠를 대표하는 핵심 개념만 남긴다.
2. **generic 단독 사용 금지**: `기초 / 실무 / 이해 / 전략 / 관리 / 분석` 같은 형용사성 키워드는 반드시 다른 명사 키워드와 결합하거나, 아예 배제한다.
3. **과특수 키워드 배제**: 전체 코퍼스에서 3개 미만 콘텐츠에만 붙을 것 같으면 넣지 않는다.

```
# 판단 인풋
title + description + original keyword_tags
→ 최종 2-3개의 reusable keywords

# generic 단독 금지 예시
❌ ["기초", "전략", "AI"]
✅ ["머신러닝", "AI"]
```

## 실제 작업 방식: 사람이 최종 판단한다

자동화에 기대고 싶은 마음은 이해하지만, 지금 단계에서는 LLM 출력을 그대로 source-of-truth에 반영하지 않는다. 다음 세 가지 인풋을 함께 보고 사람이 최종 결정한다.

| 인풋 | 역할 |
|---|---|
| `title` | 콘텐츠의 핵심 주제 파악 |
| `description` | 세부 내용 확인 |
| `original keyword_tags` | 기존 태깅과의 일관성 체크 |

배치(batch) 작업 결과는 아래 형태의 JSON으로 중간 저장한다.

```json
{
  "batch": "AI_데이터_batch03",
  "results": [
    { "cid": "content_001", "keywords": ["머신러닝", "데이터전처리"] },
    { "cid": "content_002", "keywords": ["자동화", "파이썬"] }
  ]
}
```

이 결과를 검토한 뒤 source-of-truth 파일에 반영하고, 이전 키워드와의 mismatch를 별도로 기록한다. 나중에 "왜 이 콘텐츠의 키워드가 바뀌었는가"를 추적할 수 있게.

## 마치며

"키워드를 잘 뽑는다"는 말의 의미가 달라졌다. 많이 뽑는 게 아니라, 코퍼스 전체에서 **의미 있는 교차점을 만들 수 있는** 키워드를 고르는 것. LLM은 이 판단을 보조하지만, 추천 시스템의 매칭 로직을 이해한 사람이 최종 검수해야 한다.

콘텐츠가 수백 개를 넘어가는 순간, 태깅 품질이 추천 품질을 결정한다. 그리고 태깅 품질은 키워드의 수가 아니라 **재사용성**으로 측정된다.
