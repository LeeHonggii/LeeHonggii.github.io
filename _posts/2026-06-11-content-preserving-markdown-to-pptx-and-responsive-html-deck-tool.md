# 도입 (왜 이 글)

AI에게 "이 문서로 발표자료 만들어줘"라고 하면 꽤 그럴듯한 슬라이드가 나온다. 그런데 막상 원본 문서와 대조해보면 핵심 코드 블록 하나가 빠져있거나, 표의 행이 절반으로 줄어있거나, 설명 단락이 슬쩍 압축돼 있다. AI가 "요약"을 기본값으로 삼기 때문이다.

나는 이 문제를 파이프라인 수준에서 막고 싶었다. 원본 Markdown을 요약 없이 슬라이드로 분할하고, PPTX와 반응형 HTML을 동시에 뽑아내고, 마지막에는 원본 내용이 산출물에 실제로 남아있는지 자동으로 검증하는 구조다.

## 핵심 문제: AI는 왜 내용을 버리는가

LLM(Large Language Model, 대규모 언어 모델)이 발표자료를 생성할 때 흔히 "슬라이드당 3~5 bullet"이라는 암묵적 규칙을 따른다. 원본 문서가 아무리 길어도 그 틀에 맞게 잘라 넣는다. 그 과정에서 일어나는 일은 세 가지다.

- **압축**: 두 문단을 한 줄로 합침
- **삭제**: 코드 블록, 표, 수치가 "너무 상세하다"는 이유로 제외됨
- **재해석**: 원문 표현이 아닌 모델의 언어로 바뀜

해결책은 AI에게 요약을 맡기는 대신, **구조 분할(section splitting)** 만 맡기는 것이다.

## 설계 원칙: 1:1 또는 1:N 분할, 요약 없음

```
원본 Markdown
  └── H2 섹션 A  →  슬라이드 1
  └── H2 섹션 B  →  슬라이드 2 (짧으면 1장)
                 →  슬라이드 2-3 (길면 N장으로 분리)
  └── 코드 블록  →  전용 코드 슬라이드 (잘리지 않음)
  └── 표        →  전용 표 슬라이드 (행 수 그대로)
```

핵심은 **텍스트 자르기 금지** 규칙이다. 코드 블록은 분리하더라도 내용은 그대로, 표는 행을 절대 합치지 않는다. 분량이 슬라이드 한 장을 넘으면 페이지를 추가하지 내용을 버리지 않는다.

## 구현: make_deck.py 파이프라인

아래는 실제 사용하는 CLI 호출 예시다.

```bash
python3 make_deck.py \
  --md input.md \
  --style <brand> \
  --mode both \
  --vault <vault>
```

`--mode both`를 지정하면 같은 분할 트리에서 `deck.pptx`와 `index.html`을 동시에 생성한다. 두 산출물의 소스는 `source.md`(분할된 중간 표현)로 동일하다.

**PPTX 경로**: python-pptx 라이브러리로 텍스트·코드·표를 슬라이드 오브젝트에 1:1 매핑한다. 레이아웃 자유도가 낮은 대신 사무 환경에서 바로 편집 가능한 `.pptx`가 나온다.

**HTML 경로**: 동일한 `source.md`에서 Marp(Markdown 기반 슬라이드 생성 프레임워크) 또는 직접 작성한 CSS 그리드로 반응형 HTML을 생성한다. 화면 크기에 맞게 레이아웃을 선택하는 데 초점을 두어, 발표 현장에서 브라우저만으로도 쓸 수 있다.

## 검증 단계: verify_no_loss

생성이 끝나면 원본 문서의 핵심 요소가 HTML 산출물에 실제로 있는지 검사한다.

```python
def verify_no_loss(body: str, html_text: str) -> list[str]:
    missing = []
    for block in extract_code_blocks(body):
        if block.strip() not in html_text:
            missing.append(f"코드블록 유실: {block[:40]}...")
    for row in extract_table_rows(body):
        if row.strip() not in html_text:
            missing.append(f"표 행 유실: {row[:40]}...")
    return missing
```

`extract_code_blocks`와 `extract_table_rows`는 원본 Markdown을 파싱해 검증 대상 문자열 목록을 만든다. 반환 리스트가 비어있어야 정상 통과다. 유실이 발견되면 해당 슬라이드 번호와 함께 경고를 출력하고 파이프라인을 멈춘다.

이 단계가 있으면 AI가 "최적화"라는 명목으로 내용을 조용히 지워도 즉시 잡힌다.

## 마치며

AI 기반 발표자료 생성의 진짜 문제는 품질이 아니라 **신뢰성**이다. 결과가 그럴듯해 보이기 때문에 내용이 빠졌는지 모르고 넘어간다. 파이프라인에 검증 단계를 넣으면 "생성됐다"와 "원본이 보존됐다"를 구분할 수 있게 된다.

요약하면 세 가지다.

1. AI에게 요약을 맡기지 말고 **분할만** 맡긴다.
2. PPTX와 HTML은 **같은 중간 표현**에서 뽑아 일관성을 유지한다.
3. 생성 후 **verify_no_loss**로 코드·표·핵심 문장의 존재를 자동 확인한다.

다음 단계로는 분할 단위를 H3까지 내려가게 조정하거나, 유실된 블록을 자동으로 추가 슬라이드에 붙이는 복구 로직을 붙이는 방향을 보고 있다.
