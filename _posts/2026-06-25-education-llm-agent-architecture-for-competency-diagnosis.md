# 도입 (왜 이 글)

교육 분야 LLM Agent 과제를 설계하다 보면 으레 이런 그림이 나온다. "학습자가 질문하면 Agent가 커리큘럼을 추천한다." 기능적으로는 맞다. 그런데 막상 PoC를 준비하면서 한 가지 질문이 자꾸 걸렸다.

> "이 사람이 추천받은 걸 다 배우고 나면, 우리는 그 성장을 어떻게 확인하지?"

확인 방법이 없으면 Agent는 그냥 비싼 챗봇이다. 교육 효과를 측정하지 못하면 예산 집행 근거도, 시스템 개선 신호도 없다. 이 글은 "추천에서 멈추는 Agent"를 "진단-로드맵-재측정 폐루프(closed-loop)"로 업그레이드하는 설계를 정리한 기록이다.

---

## 1. 역량 진단은 단순 퀴즈가 아니다

처음엔 "10문제 맞히면 중급"으로 접근했다. 틀렸다. 직무역량 진단에는 최소 다섯 레이어가 얽혀 있다.

| 레이어 | 역할 | 핵심 기술 |
|---|---|---|
| 스킬 추출 | JD·이력서에서 스킬 태그 파싱 | NER, 직무 온톨로지 |
| 역량 매핑 | 스킬 → 역량 프레임워크 정렬 | 역량 체계(Competency Framework) |
| 지식추적 | 학습 이력 기반 숙련도 갱신 | Knowledge Tracing (BKT/DKT) |
| 적응형 진단 | 응답에 따라 문항 동적 선택 | IRT/CAT |
| 자동채점 | 서술형·코드 채점 | AES, LLM Judge |

**IRT/CAT**는 Item Response Theory / Computerized Adaptive Testing의 약자로, 문항 난이도와 학습자 능력을 동시에 추정하면서 최소 문항으로 최대 정보를 뽑는 기법이다. 50문제짜리 시험을 15문제로 줄이면서도 신뢰도를 유지할 수 있다.

**Knowledge Tracing(지식추적)**은 "이 학습자가 이 스킬을 지금 얼마나 알고 있는가"를 시간 흐름에 따라 확률로 표현한다. BKT(Bayesian Knowledge Tracing)는 간단하지만 스킬 독립 가정이 강하고, DKT(Deep Knowledge Tracing)는 LSTM 기반으로 스킬 간 전이를 학습한다.

---

## 2. 스킬 갭 분석과 로드맵 생성

진단 결과가 나오면 다음은 "무엇을 배워야 하는가"다. 이 부분에서도 단순 추천과 구조적 추천의 차이가 크다.

**스킬 갭**은 간단히 쓰면 이렇다:

```python
skill_gap = required_skills - current_skills
```

하지만 이걸 실제 학습 경로로 바꾸려면 선수관계 그래프(prerequisite graph)가 필요하다. "Python을 모르면 pandas를 배워도 소용없다"는 의존 관계를 DAG(Directed Acyclic Graph, 방향성 비순환 그래프)로 표현하고, 갭을 위상 정렬로 순서화한다.

LLM이 이 그래프를 직접 생성하게 두면 환각(hallucination) 위험이 크다. 내가 선택한 방식은 **RAG(Retrieval-Augmented Generation, 검색증강생성)** 기반으로 사전 구축된 역량 체계 문서를 검색해 LLM이 로드맵을 "작성"하되, 선수관계는 구조화된 DB에서 제약으로 걸었다. RAG 품질 지표로는 **RAGAS faithfulness >= 85%** 를 기준선으로 잡았다. Faithfulness는 생성된 답변이 검색된 컨텍스트에 얼마나 근거하는지 측정하는 RAGAS 메트릭이다.

---

## 3. 재측정이 없으면 폐루프가 아니다

로드맵대로 학습이 끝난 뒤 다시 진단을 돌리는 것, 이게 폐루프(closed-loop)의 핵심이다. 그런데 "같은 시험 다시 보기"로는 성장 측정이 안 된다. 문항을 외웠을 수 있기 때문이다.

재측정을 설계할 때 고려한 요소 세 가지:

1. **문항 뱅크 분리** — 사전/사후 평가 문항을 겹치지 않게 관리. IRT 파라미터로 난이도를 통제해 두 세트가 동등한 난이도임을 보장.
2. **채점 일관성 검증** — 서술형 자동채점 시 평가자 간 일치도를 **Krippendorff's alpha(크리펜도르프 알파) >= 0.8** 로 관리. 이 값이 0.8 미만이면 채점 기준이 불안정하다는 신호.
3. **human-in-the-loop** — 점수 변화가 임계값 이상이거나 이하일 때 전문가가 결과를 리뷰. LLM 판단을 맹신하지 않는 안전망.

재측정 결과가 다시 지식추적 모델에 피드백되면, Agent는 다음 사이클의 로드맵을 더 정교하게 개인화할 수 있다.

---

## 4. 구조 요약: 세 단계가 하나의 루프

```
[진단]
JD/이력서 → 스킬 추출 → IRT/CAT 적응형 평가 → 역량 점수 산출
        ↓
[로드맵]
스킬 갭 = 요구 - 보유 → 선수관계 DAG 탐색 → RAG 기반 학습 경로 생성
        ↓
[재측정]
학습 완료 → 별도 문항 뱅크로 사후 평가 → Δ점수 계산 → 지식추적 갱신
        ↑_____________________________피드백 루프__________________________|
```

각 단계는 독립적으로 모듈화할 수 있다. PoC 단계라면 IRT 없이 단순 점수제로 시작하고, 채점도 LLM Judge 하나로 단순화해도 된다. 중요한 건 루프 자체를 설계에 포함시키는 것이다. 추천만 하고 끝나는 구조는 처음부터 버려야 한다.

---

## 마치며

교육 LLM Agent를 설계하면서 가장 크게 배운 점은, "무엇을 추천하느냐"보다 "추천 이후를 어떻게 닫느냐"가 시스템의 가치를 결정한다는 것이었다. 진단 없는 추천은 근거가 없고, 재측정 없는 교육은 효과를 증명할 수 없다.

스킬 갭 공식 한 줄, IRT/CAT 적응 진단, RAGAS로 RAG 품질 관리, Krippendorff alpha로 채점 신뢰도 확보 — 각 조각이 서로 물려 있을 때 비로소 "교육 AI"라고 부를 수 있겠다는 생각이 들었다.
