# 도입 (왜 이 글)

콘텐츠 메타데이터에 키워드를 붙이는 작업을 처음 시작했을 때, 나는 "많이 붙일수록 좋다"고 막연히 생각했다. LLM에 콘텐츠 설명을 던지면 10개, 20개씩 뽑아주니까 그냥 넣으면 될 것 같았다.

그게 틀렸다는 걸 깨닫는 데 꽤 오래 걸렸다.

추천 시스템(recommendation system)에서 키워드는 유저와 콘텐츠를 연결하는 다리다. 다리가 너무 촘촘하면 오히려 아무것도 연결이 안 된다. 이 글은 그 깨달음을 디버깅 기록처럼 정리한 것이다.

## 문제: LLM은 "그럴듯한 것"을 뽑지, "매칭되는 것"을 뽑지 않는다

LLM에게 콘텐츠 설명 하나를 주고 키워드를 추출하면, 문맥상 등장하는 단어는 전부 후보가 된다. 그 결과물엔 두 가지 유형의 쓰레기가 섞인다.

**너무 넓은 키워드** — `기초`, `실무`, `이해`, `전략`, `관리`, `분석`. 이런 단어는 사실상 모든 콘텐츠에 붙을 수 있어서, 추천 매칭 신호로서 가치가 0에 가깝다. 이 프로젝트에서는 아예 **generic 단독 금지 목록**을 만들어 필터링했다.

**너무 좁은 키워드** — 특정 콘텐츠 하나에서만 쓰이는 고유 표현. 이런 키워드는 비슷한 콘텐츠가 없으니 추천 그래프에서 고립된 섬이 된다. 매칭이 될 일이 없다.

결론: 키워드는 "이 콘텐츠를 설명하는 단어"가 아니라 "이 콘텐츠를 찾아올 만한 유저, 또는 함께 소비될 만한 다른 콘텐츠를 연결하는 단어"여야 한다.

## 원칙: 제목 + 설명 + 원본 태그를 같이 보고, 2-3개만 고른다

LLM을 자동 추출기로 쓰지 않고, **검수 파트너**로 운영하는 방식으로 전환했다. 프롬프트 구조는 대략 이렇다.

```json
{
  "콘텐츠 제목": "...",
  "콘텐츠 설명": "...",
  "기존 원본 키워드": ["...", "..."],
  "전체 키워드 목록": ["...", "..."]
}
```

LLM에게 주는 지시는 간단하다: *"전체 키워드 목록 안에서, 이 콘텐츠의 핵심 의미에 맞는 것 2-3개만 골라라."*

자유 추출이 아니라 **분류 체계 안에서의 선택**이다. 이렇게 하면 LLM이 없는 키워드를 만들어내는 환각(hallucination)을 막을 수 있고, 결과 키워드가 시스템 전체 어휘와 일치하게 된다.

## strand는 참고값일 뿐, 의미가 맞으면 cross-strand도 허용

콘텐츠마다 strand(주제 계열, 예: AI계열 / 데이터계열)가 붙어 있었다. 처음엔 "이 콘텐츠는 AI strand니까 AI 키워드만 써야 한다"고 엄격하게 제한했다.

그게 두 번째 실수였다.

실제로는 AI 콘텐츠인데 데이터 전처리 키워드가 더 잘 맞는 경우가 있다. strand는 편의상 분류일 뿐, 의미 매칭의 기준이 아니다. **cross-strand 키워드도 의미가 맞으면 허용한다**는 원칙을 명시하고 나서야 결과 품질이 올라갔다.

## 반영과 검증: mismatch = 0 을 기준으로

배치(batch) 단위로 결과를 뽑으면 이런 형태가 된다.

```json
{
  "batch": "AI_데이터_batch01",
  "results": [
    { "cid": "content-001", "keywords": ["AI", "데이터"] },
    { "cid": "content-002", "keywords": ["머신러닝", "분류"] }
  ]
}
```

이걸 source-of-truth 메타데이터 파일(`metadata.json`)에 `keyword_tags` 필드로 덮어쓴 뒤, 검증 스크립트로 `result.json ↔ metadata.json` 간 mismatch 개수를 확인한다. **mismatch = 0** 이 확인될 때까지 반영을 완료로 처리하지 않는다.

단순한 체크지만, 이 단계가 없으면 "뽑긴 했는데 실제 메타데이터에는 안 들어간" 상태가 조용히 쌓인다.

## 마치며

키워드 태깅은 지루해 보이지만, 추천 시스템 전체의 신호 품질을 결정한다. 많이 뽑는 것보다 잘 고르는 것, 자유 추출보다 분류 체계 안에서의 선택, 뽑은 뒤 검증까지가 한 세트다.

LLM은 이 과정에서 "자동화 도구"가 아니라 "빠른 검수 파트너" 역할이 맞다. 판단 기준은 사람이 설계하고, LLM은 그 기준 안에서 작동하게 하는 것 — 이게 이번 작업에서 얻은 가장 실용적인 교훈이다.
