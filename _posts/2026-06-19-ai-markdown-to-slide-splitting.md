# 도입 — 왜 이 글을 쓰는가

마크다운 문서를 발표 슬라이드로 바꾸는 작업을 AI에게 맡겼다. 처음엔 단순해 보였다. "이 문서를 슬라이드 10개로 나눠줘"라고 하면 되겠지 싶었다.

결과는 참담했다. AI는 원문을 **요약하거나 재작성**했다. 어떤 슬라이드에는 같은 내용이 두 번 나왔고, 어떤 단락은 조용히 사라졌다. 문서 변환이 아니라 **문서 손상**이었다.

그 원인을 파고들다 깨달은 것이 있다. AI에게 "어떤 내용을 담아라"가 아니라 "원본의 몇 번째 줄부터 몇 번째 줄까지를 써라"고 말해야 한다는 것. 라인 번호가 핵심이었다.

---

## 본문 복사 대신 범위 지정 — start_line / end_line

처음에 나는 AI에게 이렇게 요청했다.

> "각 슬라이드에 들어갈 내용을 JSON 배열로 정리해줘."

AI는 내용을 **직접 작성해서 반환**했다. 요약된 문장, 바뀐 표현, 빠진 코드 블록. 원본과 달라진 지점이 너무 많았다.

프롬프트를 바꿨다. 슬라이드 분할 결과를 `start_line`과 `end_line`으로만 표현하게 했다.

```
<<<SPLIT>>>
[
  {"title": "추천 시스템이란 — 방식 비교", "start_line": 6, "end_line": 17},
  {"title": "콘텐츠 기반 필터링 구조", "start_line": 18, "end_line": 31}
]
<<<END>>>
```

규칙도 함께 명시했다.

- **원본 텍스트 유실 0** — 모든 라인은 정확히 하나의 슬라이드에 속해야 한다
- **슬라이드 간 라인 범위 겹치지 말 것** — 같은 라인이 두 슬라이드에 중복되면 안 된다
- AI는 내용을 재작성하지 않고, 범위만 결정한다

이 구조를 도입한 뒤로 원문 유실이 사라졌다. AI가 "편집자"가 아니라 "인덱서"로 역할이 바뀌었기 때문이다.

---

## 한 슬라이드, 한 메시지 — 밀도 기준 설계

라인 번호로 범위를 잡는다고 끝이 아니다. 범위를 얼마나 좁게 잘라야 하는가가 다음 문제였다.

H2 헤딩 하나 = 슬라이드 하나라는 단순 규칙으로 시작했다. 하지만 어떤 섹션은 헤딩 하나 아래에 표, 코드 블록, 설명 단락이 함께 있었다. 이걸 한 슬라이드에 넣으면 **정보 과부하** — 청중이 읽는 데만 2분이 걸린다.

기준을 다시 세웠다.

| 조건 | 처리 |
|---|---|
| 헤딩 + 3줄 이하 단락 | 하나의 슬라이드 |
| 헤딩 + 표 | 표만 슬라이드 하나로 분리 |
| 헤딩 + 단계별 워크플로우 | 단계별로 슬라이드 분리 |
| 두 개 이상의 H3 하위 항목 | H3 단위로 분리 |

이 기준을 프롬프트에 직접 주었더니 AI가 밀도 판단을 꽤 잘 따라왔다. 단, "내용이 많으면 나눠라"라는 모호한 지시보다 **구체적인 구조 조건**을 주는 것이 훨씬 효과적이었다.

---

## 섞인 주제 분리 — 의미 단위가 경계다

가장 까다로운 케이스는 **하나의 섹션에 여러 주제가 섞인 경우**였다. 예를 들어, "시스템 개요" 하나의 헤딩 아래에 비교 표, 아키텍처 설명, 시나리오 예시가 모두 들어 있는 경우.

이때는 헤딩 기준이 아니라 **의미 단위(semantic unit)** 기준으로 자르도록 프롬프트를 설계했다.

- 비교 표 → 독립 슬라이드
- 워크플로우 단계 → 단계별 독립 슬라이드
- 시나리오 예시 → 사례별 독립 슬라이드

프롬프트에는 이렇게 명시했다.

```
하나의 슬라이드에는 하나의 의미 단위만 담는다.
의미 단위란: 비교 표, 단계별 흐름, 단일 개념 설명, 코드 예시 중 하나.
같은 헤딩 아래라도 의미 단위가 다르면 슬라이드를 나눈다.
```

이 규칙이 들어가자 AI가 헤딩 경계를 무시하고 내용의 흐름을 따라 경계를 잡기 시작했다. 처음엔 당황스러웠지만, 결과물은 훨씬 청중 친화적이었다.

---

## 마치며 — AI에게 편집을 맡기지 말고, 구조를 맡겨라

이 작업을 통해 얻은 가장 큰 교훈은 이것이다.

> AI에게 "어떻게 표현할지"를 맡기면 원문이 흔들린다. AI에게 "어디서 어디까지"를 맡기면 원문이 보존된다.

`start_line / end_line` 기반 분할은 단순한 구현 편의가 아니라 **원문 보존의 아키텍처적 선택**이다. AI를 문서 변환 파이프라인에 쓸 때, 콘텐츠 생성 역할과 구조 분석 역할을 명확히 분리하는 것이 신뢰할 수 있는 자동화의 출발점이다.
