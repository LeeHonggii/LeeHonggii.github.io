# 도입 (왜 이 글)

프롬프트 리뷰 중에 이런 예시를 발견했다.

```
입력: "수 모으기 활동"
출력: keywords=["모으기"]
```

누군가 회의에서 나온 케이스를 그대로 few-shot(소수 예시 기반 학습) 예시로 박아놓은 것이었다. 결과는 맞았다. 테스트도 통과했다. 그런데 뭔가 찝찝했다. 다음 날, 그 찝찝함의 정체를 정리해봤다.

## Few-shot은 회귀 테스트가 아니다

Few-shot이란 LLM 프롬프트 안에 입출력 예시를 몇 개 넣어 모델이 패턴을 흉내내도록 유도하는 기법이다. 예시가 잘 만들어지면 모델은 비슷한 입력에 비슷한 출력을 낸다.

문제는 여기서 시작한다. "수 모으기 활동"이 `["모으기"]`로 나와야 한다는 사실이 프롬프트 예시로 굳혀지는 순간, 그 매핑은 **회귀 테스트 케이스**가 된다. 모델은 해당 문장을 보면 그 답을 낼 확률이 높아진다. 하지만 이건 로직이 아니다. 모델이 왜 `"모으기"`를 골랐는지, 다른 표현이 들어왔을 때도 같은 기준이 작동하는지를 보장하지 않는다.

few-shot 예시 ≠ 일반화 규칙. 이 둘을 혼동하는 순간, 프롬프트는 조용히 취약해진다.

## 모호한 문장을 단정하면 어디서 터지는가

```
입력: "전체와 부분의 관계 이해"
출력: keywords=["부분", "전체"]  # 혹은 ["분수"]?
```

이 문장은 1학년 수학일 수도 있고, 4학년 분수 단원일 수도 있다. 학년·단원 정보 없이 LLM 혼자 단일 키워드를 확정하면, 그 판단은 맥락 없이 내려진 추측이다.

추출 결과가 이후 단원 매칭이나 도구 추천에 연결된다면, 잘못된 키워드 하나가 파이프라인 전체를 엉뚱한 방향으로 흘려버린다. 오류 메시지도 없이, 그냥 조용히 틀린 결과가 나온다.

이런 입력에는 `needs_clarification: true` 플래그를 반환하거나, 단원 후보 목록을 먼저 제시하는 구조가 훨씬 안전하다.

```json
{
  "keywords": [],
  "needs_clarification": true,
  "candidates": ["1학년 수와 연산", "4학년 분수"]
}
```

단정하지 않는 것도 하나의 정답이다.

## LLM을 어디에 써야 하는가

이 경험을 정리하면서 역할 분리의 원칙을 다시 세웠다.

| 역할 | 적합한 도구 |
|------|------------|
| 후보 키워드 생성 | 커리큘럼 역인덱스, 별칭 사전, 결정적 규칙 |
| 모호성 판단 | LLM |
| 최종 선택 (후보가 명확할 때) | 규칙 기반 필터 |
| 최종 선택 (후보가 복수일 때) | LLM |

LLM은 비결정적(non-deterministic)이다. 같은 입력도 실행마다 달라질 수 있다. 반면 커리큘럼 데이터는 정해져 있다. 후보 생성처럼 정답이 고정된 영역에 LLM을 쓰면, 불필요한 불확실성이 들어온다.

결정론적으로 처리할 수 있는 건 결정론적으로. LLM은 그 이후 판단에만.

## 마치며

few-shot 예시를 추가하는 건 5분짜리 작업이다. 하지만 그 예시가 "이 케이스가 해결됐다"는 착각을 심어주면, 진짜 문제인 일반화 로직 부재는 그냥 묻힌다.

프롬프트를 다시 볼 때 이 질문을 먼저 던지게 됐다. *"이 예시는 패턴을 가르치는가, 아니면 특정 답을 외우게 만드는가?"*

그 둘의 차이가 프롬프트의 수명을 결정한다.
