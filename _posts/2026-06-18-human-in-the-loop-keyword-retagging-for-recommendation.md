# 도입 (왜 이 글)

"LLM한테 키워드 뽑아달라고 하면 되지 않나?"

나도 처음엔 그렇게 생각했다. 프롬프트 하나로 콘텐츠당 키워드 10개씩 뽑으면 끝이라고. 근데 수천 건이 쌓이고 나서 추천 품질을 확인해보니 뭔가 이상했다. 매칭이 너무 흩어졌고, 비슷한 콘텐츠인데도 전혀 다른 키워드가 붙어 있었다. 3만 건 메타데이터를 직접 재검증하면서 깨달은 건 단순했다: **키워드는 많이 뽑는 게 아니라, 추천 매칭에 쓸 수 있게 정제해야 한다.**

---

## 키워드가 많으면 왜 문제인가

추천 시스템(Recommendation System — 사용자에게 관련 콘텐츠를 자동으로 제안하는 시스템)에서 키워드는 콘텐츠 간 유사도를 측정하는 핵심 신호다. 키워드가 10개면 매칭 표면이 10배로 넓어진다. 얼핏 좋아 보이지만, 실제로는 노이즈가 함께 늘어난다.

예를 들어 "파이썬으로 pandas DataFrame 병합하기"라는 콘텐츠에 `["파이썬", "pandas", "DataFrame", "merge", "데이터분석", "ETL", "프로그래밍", "입문", "실습", "코딩"]` 이렇게 붙이면 "입문" 키워드 하나 때문에 전혀 다른 난이도의 콘텐츠와 매칭된다. 키워드가 많을수록 의도하지 않은 연결이 생긴다.

**2~3개 원칙**이 나온 이유가 여기 있다. 핵심 개념만 남기면 노이즈가 줄고, 실제로 관련 있는 콘텐츠끼리만 연결된다.

---

## 원본 키워드를 보되, 걸러야 할 것들

LLM이 새로 뽑은 키워드만 믿는 것도 위험하다. 기존에 사람이 달아둔 원본 키워드에는 콘텐츠의 실제 의도가 담겨 있기 때문에 반드시 참고해야 한다. 다만 필터링 기준이 있다.

**제외해야 할 키워드 유형:**

| 유형 | 예시 | 이유 |
|------|------|------|
| 해당 콘텐츠에만 특화된 고유 표현 | "3강 심화", "특별편" | 다른 콘텐츠와 매칭 불가 |
| 너무 generic한 단독 키워드 | "학습", "강의" | 모든 콘텐츠에 붙으면 신호 없음 |
| 중복 의미 키워드 중복 | "AI", "인공지능" 동시 사용 | 같은 신호를 두 번 낭비 |

Generic(범용) 키워드는 단독으로 쓰면 구별력이 없다. "AI"만 달린 콘텐츠는 AI 관련 모든 것과 매칭되어버린다. "AI"와 "자연어처리"를 함께 쓰는 식으로 조합할 때만 의미가 생긴다.

---

## Strand는 참고값, 의미가 다르면 Cross-strand로

이번 작업에서 콘텐츠는 Strand(스트랜드 — 주제 계열 또는 커리큘럼 묶음)로 분류되어 있었다. 초기 설계는 "같은 strand면 같은 키워드 풀에서 고른다"였는데, 실제로 작업해보니 strand 라벨이 의미를 다 설명하지 못하는 경우가 꽤 있었다.

"AI_데이터" strand인데 내용은 통계 기초인 경우, "데이터" 키워드만 달면 AI 콘텐츠와 잘못 매칭된다. 이럴 때는 cross-strand 키워드 — 즉 다른 strand에서 쓰는 키워드를 가져와서 — 실제 내용을 반영해야 한다. Strand는 분류 힌트일 뿐이고, 최종 키워드는 콘텐츠의 실제 의미에서 나와야 한다.

---

## 3만 건을 안전하게 처리하는 배치 흐름

규모가 커지면 검증 체계가 없으면 금방 망가진다. 내가 쓴 흐름은 이렇다.

**배치 파일 구조:**

```
rematch_batches/<strand>_batchNN.json
→ 작업 완료 후
<strand>_batchNN_result.json
```

result 파일의 구조는 단순하게 유지했다:

```json
{
  "batch": "AI_데이터_batch03",
  "results": [
    { "cid": "c_0412", "keywords": ["머신러닝", "데이터전처리"] },
    { "cid": "c_0413", "keywords": ["AI", "자동화", "파이프라인"] }
  ]
}
```

결과 파일이 생성되면 source-of-truth(원본 메타데이터 파일)에 `keyword_tags`를 덮어쓴다. 그 다음 반드시 확인하는 지표가 하나다:

```
mismatch = 0
```

result에 있는 cid가 source-of-truth에 반영됐는지, 키워드 값이 일치하는지 검증한다. 이게 0이 아니면 배치를 넘어가지 않는다. 세션이 끊기거나 컨텍스트가 압축되어도 이 숫자 하나로 작업 상태를 바로 파악할 수 있었다.

---

## 마치며

LLM 보조 태깅 작업에서 가장 크게 배운 건 "자동화가 사람의 판단 기준을 대체하지 않는다"는 점이다. LLM은 키워드 후보를 빠르게 제안해주지만, 추천 매칭성이 있는 키워드인지, generic 단독인지, cross-strand가 필요한지는 여전히 사람이 기준을 세워야 한다.

2~3개 원칙, generic 단독 금지, strand 참고 후 의미 우선, 그리고 mismatch=0 검증 — 이 네 가지 기준이 있으면 3만 건도 흔들리지 않고 끝낼 수 있다. 규모가 커질수록 기준이 먼저다.
