# 도입 (왜 이 글)

AI로 기술 제안서 초안을 잡는 일이 이제는 자연스럽다. 프롬프트 한 번에 10페이지 분량의 구조가 나오고, 그럴듯한 수치와 서비스명이 빼곡하게 채워진다. 문제는 그 순간부터 시작된다.

LLM 기반 서비스 관련 제안서를 검토하다가, AI가 채워 넣은 지표 목표치와 기술 서비스 가용성 정보 상당수가 공식 문서와 다르거나 아예 확인 자체가 안 된다는 사실을 마주했다. 그 경험이 이 글의 출발점이다.

## AI 초안이 위험해지는 순간

AI가 생성한 문서에서 특히 조심해야 할 항목이 있다.

| 항목 유형 | 위험도 | 이유 |
|---|---|---|
| 수치 목표치 (정확도 %, QWK 등) | 높음 | 모델이 그럴듯한 숫자를 "추정"해서 생성함 |
| 서비스명·버전 | 높음 | GA 상태, 출시 일정이 자주 바뀜 |
| 지원 언어·리전 가용성 | 높음 | 모델 컷오프 이후 변경 사항을 모름 |
| 논문·데이터셋 인용 | 중간 | 존재하지 않는 논문이 hallucination으로 생성되기도 함 |
| 일반 방법론 설명 | 낮음 | 널리 알려진 개념은 대체로 안전 |

**held-out 검증셋** — 모델 학습에 사용하지 않고 성능 평가에만 쓰는 별도 데이터셋 — 같은 개념조차, AI가 구체적 수치("held-out 셋 기준 QWK 0.82 달성 사례 존재")를 스스로 붙이면 출처 없이 문서에 넣을 수 없다. 수치가 맞더라도, 근거 없는 주장은 제출 문서에서 지뢰가 된다.

## 검증 워크플로: 작성과 확인을 분리하라

AI 초안을 실제 제출 가능한 문서로 바꾸는 핵심은 **작성 단계와 검증 단계를 분리**하는 것이다.

**1단계 — 클레임 목록 뽑기**

초안을 읽으면서 "이건 주장인가?"라고 물을 수 있는 모든 문장을 따로 표시한다. 수치, 서비스명, 기능 지원 여부, 실적 클레임이 여기에 해당한다.

**2단계 — 클레임 상태 분류**

```
교차확인 ○  — 2개 이상의 1차 자료(공식 문서, 논문, 보도자료)로 확인됨
단일출처    — 1개 문서에서만 확인, 추가 검증 필요
미확보      — 아직 공식 자료를 찾지 못함
```

이 세 상태를 각 클레임 옆에 표시해두면 어떤 주장이 제출 가능한지 한눈에 보인다. 특히 공식 문서 URL 기준으로 **리전 가용성, GA(General Availability, 정식 출시) 상태, 파인튜닝(fine-tuning, 특정 도메인에 맞게 모델을 추가 학습하는 것) 가능 여부**를 직접 확인하는 것이 핵심이다. 모델의 학습 데이터 컷오프 이후 변경된 정보는 AI가 알 수 없다.

**3단계 — 미확보 클레임 처리**

미확보 상태로 남은 항목은 두 가지 선택지뿐이다.

- 문장을 명시적으로 완화한다: "~로 예상됨 (공식 확인 필요)"
- 해당 클레임 자체를 제거하고 검증된 정보만 남긴다

"검토자가 모르겠지"라며 미확보 수치를 그냥 두는 건 절대 하지 말아야 한다.

## 자료맵: 다음 작성을 위한 인프라

검증 과정에서 모은 자료를 곧바로 문서에 녹이기보다, **파일별 자료맵**으로 먼저 저장해두면 이후 작성이 훨씬 빠르다. 구조는 단순하다.

```
이건 어디 있고  → 출처 URL / 문서명
내용은 무엇이며 → 핵심 내용 1-2줄
이렇게 작성하면 어떨까요 → 제안 문장 초안
```

예를 들어 특정 클라우드 AI 서비스의 한국어 지원 여부를 조사했다면:

```
출처: [공식 문서 URL]
확인 내용: 한국어(ko) GA 지원, 특정 리전 한정, 2024-Q3 기준
제안 문장: "해당 서비스는 공식 문서 기준 한국어 GA를 지원하며,
            현재 지정 리전에서 이용 가능합니다."
```

이렇게 해두면 문서 업데이트 시 어디를 다시 확인해야 하는지 명확해진다. RAG(Retrieval-Augmented Generation, 외부 문서를 검색해 응답에 활용하는 방식)를 연결하든 수동으로 관리하든, 이 구조는 동일하게 작동한다.

## 마치며

AI는 문서의 뼈대를 세우는 데 탁월하다. 하지만 그 뼈대에 살을 붙이는 것, 특히 수치·서비스 가용성·실적 클레임처럼 시간과 지역에 따라 달라지는 정보는 1차 자료를 직접 확인해야 한다.

초안을 의심하는 것은 AI를 불신하는 태도가 아니다. 오히려 AI를 제대로 쓰는 방법이다. 모델은 그럴듯한 패턴을 생성할 수 있지만, 공식 문서 URL 한 줄의 무게를 이길 수 없다. 검증 워크플로를 한 번 만들어두면, 다음 제안서부터는 초안 작성보다 검증이 더 빠르게 돌아간다는 걸 직접 경험하게 된다.
