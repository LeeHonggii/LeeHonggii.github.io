# 도입 — 왜 이 글을 쓰는가

어느 날 갑자기 용어사전 생성 기능만 멈췄다. 다른 AI 기능들은 멀쩡히 돌아가고 있었다. API 키를 의심했고, 네트워크 문제를 의심했고, 내 코드를 의심했다. 범인은 그 어디에도 없었다. 결국 찾아낸 원인은 단 하나 — **모델명 하나가 퇴역(retired)되어 있었다.**

이 글은 "기능 일부만 멈출 때 어디서부터 확인할 것인가"에 대한 디버깅 기록이다.

---

## 증상: 기능마다 다른 제공자를 쓰고 있었다

이 서비스는 생성 기능이 여러 개다. 요약, 번역, 태그 추출, 용어사전 생성 등. 처음엔 모두 같은 LLM 제공자(provider)를 쓴다고 막연히 생각했는데, 코드를 펼쳐보니 사정이 달랐다.

| 기능 | 제공자 |
|---|---|
| 요약, 번역 | OpenAI |
| 태그 추출 | OpenAI |
| **용어사전 생성** | **Google Gemini** |

기능마다 제공자가 달랐고, 자연스럽게 모델명도 달랐다. OpenAI 기능이 잘 된다고 해서 Gemini 쪽 인증이나 네트워크가 문제 없다는 뜻은 아니었다. 당연한 말 같지만, 장애 상황에서는 "다른 기능이 되니까 인프라 문제는 아니겠지"라는 착각에 빠지기 쉽다.

---

## 가설 좁히기: API 키인가, 모델인가

첫 번째 가설은 API 키 만료 또는 권한 문제였다. 확인은 간단하다. 실제 호출을 직접 날려보면 된다.

```bash
# 단순 curl로 모델 접근 가능 여부 확인
curl -X POST \
  "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=$GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"contents":[{"parts":[{"text":"Hello"}]}]}'
```

응답은 `404 Not Found`였다. API 키 문제라면 `401 Unauthorized`나 `403 Forbidden`이 떴을 것이다. **404는 모델 자체가 존재하지 않는다는 뜻**이다.

Google은 주기적으로 구형 모델을 퇴역시킨다. `gemini-2.0-flash`와 `gemini-2.0-flash-lite`가 그 대상이었다. 코드가 문제가 아니라, 내가 부르고 있는 모델이 이미 사라진 상태였다.

---

## 수정: 모델명 교체와 하드코딩의 대가

수정 자체는 간단했다.

```
gemini-2.0-flash      →  gemini-2.5-flash
gemini-2.0-flash-lite →  gemini-2.5-flash-lite
```

문제는 이 이름이 **여러 파일에 하드코딩**되어 있었다는 점이다. `ripgrep`으로 범위를 먼저 파악했다.

```bash
rg "gemini-2.0-flash"
```

설정 파일, 유틸 모듈, 서비스 클래스 — 세 군데에 흩어져 있었다. 하나씩 찾아서 고쳤지만, 만약 중앙 상수 파일 하나에 모델명이 모여 있었다면 수정은 한 줄로 끝났을 것이다.

---

## 교훈: 모델 수명주기를 인프라처럼 관리해야 한다

이번 장애에서 얻은 교훈 세 가지를 정리한다.

**1. 기능별 제공자와 모델명을 먼저 비교하라.**
여러 기능 중 일부만 멈추면, 멈춘 기능이 다른 제공자나 다른 모델을 쓰는지부터 확인한다. "다른 기능이 된다"는 사실은 공통 인프라 문제를 배제할 뿐이다.

**2. 404 응답은 키 문제가 아니라 모델 퇴역일 수 있다.**
LLM API에서 `404`가 나오면 모델명 자체를 의심해야 한다. curl 한 줄로 해당 모델이 살아있는지 바로 확인할 수 있다.

**3. 모델명은 반드시 중앙 설정에서 관리하라.**
`MODEL_GEMINI_FLASH = "gemini-2.5-flash"` 같은 상수를 한 곳에 두면, 다음 퇴역 때 수정 범위가 한 줄로 줄어든다. 모델 제공자들은 보통 퇴역 일정을 미리 공지하지만, 실제로 그 공지를 놓치는 경우가 훨씬 많다.

---

## 마치며

코드는 아무것도 바뀌지 않았는데 기능이 멈추는 경험은 낯설다. LLM을 외부 서비스로 사용하는 이상, 그 서비스의 수명주기가 곧 내 코드의 수명주기에 영향을 미친다. 모델 퇴역은 버그가 아니라 **외부 의존성 변경**이다. 라이브러리 버전 관리하듯, 모델명도 추적 가능한 상태로 관리해야 한다는 것을 이번에 몸으로 배웠다.
