# 도입 (왜 이 글)

처음엔 빠르게 돌아갔다. 프롬프트에 예시 몇 개를 박아 넣으니 LLM이 원하는 키워드를 뽑아냈다. "이차방정식 풀기" → `quadratic_equation`, "미적분 기초" → `calculus_intro`. 몇 가지 케이스를 few-shot(소수 예시 학습)으로 고정하자 회귀 테스트도 통과했다.

문제는 실제 사용자가 그 예시처럼 입력하지 않는다는 것이었다.

"이차방정식 어떻게 풀어요", "2차 방정식 개념", "근의 공식 뭐야" — 같은 의도인데 표면이 전부 달랐다. few-shot이 커버하지 못한 변형은 엉뚱한 키워드로 추출되거나 아예 매핑에 실패했다. 이 글은 그 한계를 직면하고, 로그 기반 정규화로 전환한 판단 과정을 정리한 회고다.

---

## Few-Shot의 함정: 예시는 회귀 방어일 뿐이다

few-shot은 특정 입력-출력 쌍을 LLM에게 "이렇게 해" 하고 보여주는 방식이다. 내가 쓰던 패턴은 대략 이랬다:

```text
예시 1: "이차방정식 풀기" → keywords: ["quadratic_equation"]
예시 2: "미적분 기초 개념" → keywords: ["calculus_intro"]
예시 3: "확률 기본 문제" → keywords: ["probability_basic"]
...
```

이 방식이 잘 작동하는 상황은 딱 하나다. **이미 본 케이스가 다시 들어올 때.** 기존 예시와 비슷한 표현이면 LLM이 패턴을 흉내내서 맞춘다. 그래서 회귀 테스트를 막는 데는 효과적이다.

그런데 일반화(generalization)에는 취약하다. 어미 하나만 달라져도 ("풀기" → "풀어줘"), 띄어쓰기가 붙어도 ("이차방정식" → "이 차 방정식"), 학년 수식어가 붙어도 ("중학교 2학년 이차방정식") — 예시에 없는 변형은 LLM이 추측으로 때운다. 추측은 종종 틀린다.

더 나쁜 건, 이 실패가 조용히 일어난다는 점이다. 에러가 나지 않고 그냥 다른 키워드가 나온다.

---

## 로그를 열어보니: 사용자는 훨씬 다양하게 입력했다

실제 파이프라인에 쌓인 개발 로그 **2,104건**을 꺼냈다. 목적은 단순했다 — 사람들이 실제로 어떻게 입력하는지 패턴을 보는 것.

```bash
python scripts/analyze_user_inputs_for_normalization.py
```

스크립트를 돌리자 중복 제거(dedupe) 후 **1,849건**이 남았다. 255건은 완전히 동일한 문자열이었다. 남은 1,849건을 훑으니 변형의 종류가 보이기 시작했다:

| 변형 유형 | 예시 |
|---|---|
| 어미·조사 변형 | "풀기" / "풀어줘" / "푸는 방법" |
| 띄어쓰기 불일치 | "이차방정식" / "이 차 방정식" |
| 오타 | "확률" / "확율" / "확뉼" |
| 학년·차시 수식어 | "중2 이차방정식" / "고1 수1 미적분" |
| 동의어·alias | "근의 공식" / "이차방정식 해" / "판별식" |

이걸 보고 나서야 실감했다. **이 문제는 LLM 문제가 아니라 정규화(normalization) 문제였다.** 입력을 표준 형태로 만들기 전에 LLM에 던지고 있었던 것이다.

---

## 임베딩 이전에 먼저 해결할 수 있는 것들

흔히 이런 상황에서 "임베딩(embedding) 유사도로 처리하면 되지 않나?" 하는 생각이 든다. 나도 잠깐 그 방향을 봤다. 하지만 임베딩은 비용이 따른다 — 추론 시간, 모델 의존성, 그리고 유사도 임계값을 어디에 잡을지 결정해야 하는 모호성.

그 전에 규칙 기반으로 해결할 수 있는 레이어가 있었다.

**1. 텍스트 정규화**

```python
def normalize_text(s: str) -> str:
    s = s.strip().lower()
    s = re.sub(r"\s+", " ", s)          # 띄어쓰기 정리
    s = unicodedata.normalize("NFC", s) # 유니코드 정규화
    return s

def normalize_for_match(s: str) -> str:
    s = normalize_text(s)
    s = re.sub(r"[^\w가-힣]", "", s)    # 특수문자 제거
    return s
```

단순해 보이지만, 이 두 함수로 표면 변형의 상당수가 정리됐다.

**2. Alias 테이블**

"근의 공식", "이차방정식 해", "판별식" 같은 동의어는 alias 테이블로 연결했다. 로그에서 빈도 높은 표현을 canonical(표준) 키워드로 정하고, 나머지를 alias로 등록하는 방식이다.

**3. Curriculum 역인덱스(inverted index)**

"중2 수학 1학기 3단원"처럼 학년·학기·단원이 포함된 입력은 별도로 파싱해서 curriculum 역인덱스에서 직접 조회하도록 분리했다. 이 경로는 LLM을 거치지 않는다.

**4. 모호성 게이트(ambiguity gate)**

정규화 후에도 여러 키워드에 매핑될 수 있는 경우는 LLM에 넘기기 전에 명시적으로 플래그를 달아 처리 경로를 달리했다.

---

## 설계를 바꾸고 나서 달라진 것

가장 크게 달라진 건 **LLM이 처리해야 할 입력의 범위**가 줄었다는 점이다. 정규화·alias·역인덱스로 해결되는 케이스는 LLM에 도달하지 않는다. LLM에게 남겨진 것은 진짜로 언어 이해가 필요한 케이스들이다.

그 결과로 few-shot 예시도 다시 쓰게 됐다. 이제 예시는 "표준 입력 → 키워드" 매핑이 아니라, "정규화가 끝난 후에도 여전히 판단이 필요한 모호한 입력"을 다루는 예시로 바뀌었다. 역할이 달라진 것이다.

---

## 마치며

few-shot은 버려야 할 기술이 아니다. 다만 내가 그것을 **정규화의 대체제**로 쓰고 있었다는 게 문제였다. 예시를 더 추가하면 커버리지가 늘 것이라고 착각했지만, 사용자 입력의 표면 변형은 예시로 따라잡을 수 있는 범위를 넘어선다.

실제 로그를 열어보지 않았다면 계속 프롬프트를 다듬고 있었을 것이다. 데이터를 먼저 보는 것, 그게 시작이었다.
