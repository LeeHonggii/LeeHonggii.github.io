# 도입 (왜 이 글)

"정규화 로직을 어떻게 짜야 하지?" 이 질문에 처음엔 막연하게 답하려 했다. 형태소 분석기? 임베딩 유사도? 그런데 잠깐 — 실제 사용자들이 어떻게 입력하는지 한 번이라도 제대로 봤던가?

결국 로그 파일을 꺼냈다. 원본 2,104건. 중복 제거 후 1,849건. 이 숫자들을 직접 들여다보고 나서야, 내가 풀려던 문제의 실체가 보였다.

## 로그를 열기 전 가졌던 가설들

임베딩 기반 유사도 검색을 쓰면 표현이 달라도 알아서 매칭되지 않을까, 라고 생각했다. "이차방정식"이든 "이차 방정식"이든 벡터 공간에서 가깝겠지, 하고.

그런데 분석 스크립트를 돌려보니 현실은 달랐다.

```bash
python scripts/analyze_user_inputs_for_normalization.py
```

출력 결과를 훑으면서 실제 변형 패턴을 분류해봤다.

| 변형 유형 | 예시 | 빈도 |
|---|---|---|
| 띄어쓰기 불일치 | `이차방정식` / `이차 방정식` | 높음 |
| 요청문 후미 포함 | `인수분해 알려줘`, `미적분 문제 풀어줘` | 중간 |
| 학년·차시 표기 혼재 | `중2-1`, `중학교 2학년 1학기` | 중간 |
| 오타 | `뺄샘`, `팬투미노` | 낮음 |
| 어미 변형 | `함수의 뜻`, `함수란` | 낮음 |

임베딩으로 해결할 문제가 아니었다. **표면 변형(surface variation)** — 즉 같은 단어를 다르게 쓰는 패턴이 압도적이었다.

## 안전한 정규화부터 쌓아 올리기

정규화 함수를 두 층으로 나눴다. 먼저 부작용 없이 항상 적용해도 되는 것들:

```python
def normalize_text(text: str) -> str:
    text = unicodedata.normalize("NFC", text)   # 유니코드 정규형 C
    text = text.strip()
    text = re.sub(r"\s+", " ", text)             # 연속 공백 → 단일 공백
    text = re.sub(r"(알려줘|설명해줘|풀어줘|가르쳐줘)$", "", text).strip()
    return text
```

NFC(Normalization Form Canonical Composition) — 유니코드에서 같은 문자를 다른 코드포인트로 표현하는 경우를 통일하는 방식. 한국어 입력에서 자음·모음 분리 문제를 막아준다.

매칭용 정규화는 여기서 한 단계 더 간다:

```python
def normalize_for_match(text: str) -> str:
    text = normalize_text(text)
    text = re.sub(r"[0-9]+학년\s*[0-9]+학기", _normalize_grade, text)
    return ALIAS_MAP.get(text, text)
```

`ALIAS_MAP`은 오타·비표준 표기를 표준형으로 매핑하는 딕셔너리다:

```python
ALIAS_MAP = {
    "뺄샘": "뺄셈",
    "팬투미노": "펜토미노",
    "이차방정식의근": "이차방정식의 근",
}
```

조사 제거(`은/는/이/가`)와 오타 자동보정은 일단 보류했다. "함수가"에서 "가"를 지우는 건 쉬워 보이지만, "가우스"처럼 어미처럼 끝나는 고유명사와 충돌할 수 있다. 보수적으로 가기로 했다.

## 중등 매핑 실패 — 정규화 문제가 아니었다

로그 분석 중 중학교 단원 매핑 실패 케이스가 유독 많았다. 처음엔 "중등 용어가 정규화 사전에 없는 것 아닐까" 의심했다.

파고들어 보니 원인은 달랐다. DB 역조회 쿼리에서 `school_level` 컬럼을 필터링하지 않아, 초등 커리큘럼 인덱스가 중등 질의에 먼저 걸리는 버그였다.

```python
# 버그 있던 쿼리 (school_level 미사용)
results = curriculum_index.search(normalized_query)

# 수정 후
results = curriculum_index.search(normalized_query, school_level=detected_level)
```

정규화 로직을 계속 고쳐봤자 이 버그는 안 잡혔을 것이다. 로그를 분류해서 **어디서 실패하는지** 먼저 확인한 덕분에 삽질을 줄였다.

## 마치며

"감으로 설계하지 말자"는 말은 많이 들었지만, 실제로 로그 2천 건을 앉아서 분류해본 건 처음이었다. 그러고 나니 설계 방향이 훨씬 명확해졌다.

정규화는 복잡도 순서대로 쌓는 게 맞다. NFC → 공백 → 요청문 후미 → 학년 표기 → 별칭 맵. 형태소 분석이나 임베딩은 이 단순한 레이어들이 감당 못 하는 케이스가 실제로 많을 때 꺼내도 늦지 않다.

그리고 매핑 실패를 만나면 정규화보다 먼저 데이터 흐름을 의심하자. 내 경우엔 `school_level` 하나가 빠진 게 문제였다.
