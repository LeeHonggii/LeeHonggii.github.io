# 도입 (왜 이 글)

추천 시스템을 처음 붙였을 때는 잘 돌아갔다. LLM(Large Language Model, 대규모 언어 모델)이 JSON을 뱉고, DB에서 메타데이터를 꺼내고, 후보 리스트를 만들었다. 그런데 어느 날부터 결과가 이상했다. 에러 로그도 없고, 예외도 없고, 그냥 추천이 빈 배열로 돌아왔다. 조용히 망가져 있었다.

원인을 파고들어 보니 세 군데에서 각각 다른 방식으로 데이터가 깨져 있었다. LLM 응답, DB JSON 컬럼, 후보 리스트 — 셋 다 "정상처럼 보이지만 사실은 아닌" 상태였다.

## LLM JSON 응답: dict가 아닐 수 있다는 전제

`parse_json_response`가 항상 `dict`를 반환할 거라고 믿었다. 함수 이름부터 그렇게 생겼으니까. 그런데 LLM이 JSON 포맷을 어긴 응답을 내놓거나, 파싱 라이브러리가 빈 문자열을 받으면 `None`이 돌아온다. `list`가 돌아오는 경우도 있었다.

```python
parsed = parse_json_response(llm_output)

# 이 한 줄이 없으면 아래 .get() 에서 AttributeError
if not isinstance(parsed, dict):
    return fallback
```

이 체크 없이 `parsed.get("keywords")` 를 호출하면, `parsed`가 `None`이거나 `list`일 때 `AttributeError`가 터진다. 더 나쁜 건 이 예외가 상위에서 묵살되면서 빈 결과만 반환되는 경우다. 에러도 없이 조용히 실패한다.

LLM 응답은 외부 시스템 경계(external boundary)다. 내부 코드처럼 신뢰하면 안 된다.

## DB JSON 컬럼: 비문자열 값이 섞이면 문자열 연산이 터진다

DB에 JSON 배열로 저장된 키워드 컬럼을 읽어서 소문자 변환과 부분 문자열 매칭을 돌렸다. 대부분은 잘 됐다. 그런데 어떤 행에는 `null`이 들어 있거나, 정수가 섞여 있었다.

```python
# 이 코드는 int나 None이 섞이면 TypeError
for kw in db_keywords:
    if query.lower() in kw.lower():  # kw가 int면 AttributeError
        ...
```

방어 코드는 단순하다.

```python
keywords = {k for k in raw_keywords if isinstance(k, str) and k.strip()}
```

한 줄로 비문자열과 빈 문자열을 모두 걸러낸다. DB에서 온 데이터도 외부 경계로 취급해야 한다. 스키마가 있어도 실제 데이터는 그 스키마를 항상 따르지 않는다.

## 리뷰 지적 → 실패 테스트 먼저, 수정 나중

코드 리뷰에서 "이 경우 터질 것 같다"는 지적을 받으면 반사적으로 바로 고치고 싶어진다. 그런데 그렇게 하면 같은 버그가 나중에 다른 형태로 다시 들어올 수 있다.

더 나은 흐름은 이렇다:

1. 리뷰 지적을 받는다.
2. **먼저** 그 케이스를 재현하는 실패 테스트를 작성한다.
3. 테스트가 실패하는 걸 확인한다 (`pytest tests/ -v`).
4. 그 다음 코드를 수정한다.
5. 테스트가 통과하면 커밋한다.

이렇게 하면 수정이 정말로 그 케이스를 커버하는지 증명이 되고, 나중에 리팩터링할 때 회귀(regression)를 막아주는 안전망이 생긴다. 테스트 없는 수정은 "아마 고쳐졌겠지"라는 믿음에 불과하다.

## 세 곳의 공통점: 신뢰 경계를 잘못 그었다

정리하면 문제의 뿌리는 하나다. 시스템 내부처럼 믿어야 할 곳과 외부 경계처럼 방어해야 할 곳을 구분하지 않았다.

| 위치 | 무엇을 믿었나 | 실제 위험 |
|------|-------------|-----------|
| LLM 응답 | `parse_json_response`는 항상 dict | None, list, 빈 값 가능 |
| DB JSON 컬럼 | 배열 안은 모두 문자열 | null, int 혼재 가능 |
| 후보 리스트 | 파이프라인이 항상 채워준다 | 상위 단계 실패 시 빈 배열 전달 |

신뢰 경계(trust boundary)를 넘어오는 데이터는 반드시 타입과 형태를 검증해야 한다. 검증 비용은 싸고, 조용한 실패의 디버깅 비용은 비싸다.

## 마치며

가장 무서운 버그는 예외를 던지지 않는 버그다. 추천 결과가 그냥 빈 배열로 오면 사용자는 "결과가 없나보다" 하고 넘어가고, 개발자는 한참 뒤에야 알아챈다.

LLM과 DB는 내가 직접 제어하지 않는 외부다. 거기서 오는 데이터는 항상 `isinstance` 체크와 타입 필터를 달아두자. 그리고 리뷰 지적은 테스트로 먼저 못 박아두자. 그 습관이 쌓이면 시스템이 조용히 망가지는 빈도가 눈에 띄게 줄어든다.
