# 도입 (왜 이 글)

키워드 추출 파이프라인을 만들다가 이런 경험을 했다. LLM이 "수 모으기 활동"이라는 문장에서 `["모으기"]`만 뽑아냈다. 수학 맥락에서 의미 있는 키워드는 "수 모으기"인데, 모델은 동사만 집어간 것이다. 당연히 프롬프트에 예시를 추가했다.

```
예시:
- "수 모으기 활동" → ["수 모으기"]
```

다음 테스트는 통과했다. 그런데 나는 이게 **해결**인지 **땜질**인지 구분하지 못하고 있었다.

## few-shot이 왜 착각을 부르는가

few-shot (몇 가지 예시를 프롬프트에 포함해 LLM이 패턴을 따르게 하는 방식) 은 빠르고 효과적인 도구다. 특정 입력에 대한 원하는 출력을 보여주면 모델이 그 패턴을 흉내 낸다.

문제는 모델이 **이 예시로부터 규칙을 추론하는 것이 아니라** 형태를 모방한다는 점이다.

- `"수 모으기 활동"` → 통과
- `"수 가르기 놀이"` → 다시 `["가르기"]`

두 문장의 구조는 거의 같다. 하지만 프롬프트 예시에 없으니 모델은 이전 패턴으로 돌아간다. 예시 하나가 해결한 것은 그 문장 하나뿐이었다.

## 더 위험한 케이스: 모호한 문장의 단정 매핑

더 조심해야 할 상황이 있다. 학습 목표 문장이 여러 단원에 걸쳐 쓰이는 경우다.

```
"전체와 부분의 관계 이해" → keywords=["분수"]
```

초등 저학년 맥락에서는 이 문장이 분수 단원보다 **도형의 부분/전체** 단원에 가깝다. 그런데 프롬프트 예시가 `분수`로 고정되어 있으면, 모델은 학년 컨텍스트를 고려하지 않고 분수를 뱉는다. 프롬프트에 예시를 넣는 순간, 그 문장의 해석을 **하드코딩**한 셈이 된다.

이런 모호한 문장에 대해선 오히려 `needs_clarification: true`를 반환하게 하고 상위 로직에서 처리하는 방식이 훨씬 안전하다.

```json
{
  "keywords": [],
  "needs_clarification": true,
  "reason": "학년·단원 컨텍스트 없이 단정 매핑 불가"
}
```

## 프롬프트 예시 vs 회귀 테스트, 어디에 두어야 하나

지금 돌이켜 보면, 내가 프롬프트에 박아 넣던 예시들은 두 종류가 섞여 있었다.

| 예시 유형 | 적합한 위치 | 이유 |
|---|---|---|
| 정책 설명 ("동사 단독은 키워드가 아니다") | 프롬프트 | 일반화 가능한 규칙 |
| 회의 케이스 ("수 모으기 활동" → ...) | 회귀 테스트 | 해당 문장 재현용 |
| 표면 변형 처리 ("더하기" = "덧셈") | 데이터 기반 alias | 프롬프트로 커버 불가 |

프롬프트는 **정책**을 설명하는 곳이다. "모델이 이런 패턴으로 판단하길 원한다"는 의도를 전달해야지, 특정 문장의 정답을 외우게 하는 곳이 아니다.

회의 케이스는 회귀 테스트(regression test) 로 옮겨야 한다. 그래야 나중에 프롬프트를 수정했을 때 **이 케이스가 여전히 통과하는지** 자동으로 검증할 수 있다. 프롬프트 안에 있으면 변경할 때마다 수동으로 눈으로 확인해야 한다.

표면 변형 — `더하기/덧셈/+`, `나누기/나눗셈/÷` 같은 alias — 은 프롬프트 예시 열 개로 커버할 수 없다. 이건 데이터로 관리해야 한다.

## 마치며

"이 문장이 왜 틀렸지? → 예시 추가 → 통과 → 완료"

이 흐름이 얼마나 익숙한지 돌아보게 됐다. few-shot은 강력하지만, 그 안에 넣은 예시는 **일반화된 해결책이 아니라 그 문장에 대한 정답지**다. 비슷한 문장은 여전히 틀릴 수 있고, 학년이나 단원 맥락이 달라지면 그 정답지 자체가 독이 된다.

회의 케이스를 만나면 먼저 물어봐야 할 것 같다. "이걸 프롬프트에 박는 게 맞나, 아니면 테스트로 남기고 데이터 쪽을 고쳐야 하나?"
